Design Philosophy

Progressive Disclosure：作为设计哲学的渐进呈现

Progressive Disclosure（渐进呈现）不只是一种 UI 模式 — 它是一种跨越认知科学、信息架构、产品策略和叙事设计的信息控制哲学。其核心命题是：复杂性本身不是问题，让人在错误的时刻面对复杂性才是问题。从 1985 年 Apple HIG 的萌芽[4]到 Jakob Nielsen 在 2006 年的系统化[1]，渐进呈现已经从界面设计原则扩展为产品思维的底层范式。本文从认知科学出发，逐层解析这一哲学在信息架构、产品策略、叙事教育和 UI 控件五个层次的运作机制，并提供可操作的设计指南和反模式警示。

01 / Cognitive Foundations

认知科学基础：为什么渐进呈现有效

渐进呈现的有效性建立在四项认知心理学原理之上。理解这些原理，才能在设计中做出正确的信息分层决策，而非机械套用模式。

Miller's Law

工作记忆容量有限

George Miller 在 1956 年发现，人类工作记忆平均可保持 7±2 个信息块（chunks）[5]。后续研究（Cowan, 2001）将这个数字修正为 3–5 个[6]。

对设计的启示：界面同时呈现的信息若超出工作记忆容量，用户将无法有效比较和决策。渐进呈现通过分层降低同时可见的信息量，使其保持在认知带宽之内。

Hick's Law

选项越多决策越慢

Hick 和 Hyman（1952）证明：决策时间与选项数量的对数成正比 — RT = a + b · log₂(n)[7]。10 个选项比 2 个选项的决策时间增加约 3.3 倍。

对设计的启示：减少首屏选项数量直接降低用户的决策延迟。渐进呈现将高频选项放在首层，低频选项推迟到次级界面。

Cognitive Load Theory

认知负荷理论

Sweller（1988）将认知负荷分为三类[8]：内在负荷（任务本身复杂性）、外在负荷（界面设计引入的干扰）和相关负荷（用于理解和学习的认知资源）。

渐进呈现直接削减外在认知负荷 — 隐藏当前任务不需要的信息，释放认知资源用于高效决策。

Chunking

信息分块

分块（chunking）将离散信息组织为有意义的组[9]。例如电话号码 19195552743 分块为 (919)-555-2743 后，认知负荷从 11 个数字降为 3 个信息块。

渐进呈现是分块在交互层面的应用：将功能分为「核心」和「高级」两组，每组内部再按逻辑分块，使用户在每个层级都面对可管理的信息量。

02 / Definition & Taxonomy

定义、历史与三种分类

从 1985 年的萌芽到 2006 年的系统化，渐进呈现经历了从经验直觉到设计原则的演进。当代实践将其细分为三种互补的策略模式。

1985 — 萌芽：Kristina Hooper Woolsey 在 Apple Human Interface Group 提出选择性信息披露的设计理念，主张"在界面设计中必须仔细考虑如何选择性地向用户呈现系统信息，提供精心挑选的信息片段以构建对系统的整体理解"[4]。这段论述出现在 Norman & Draper 编辑的《User Centered System Design》（1986）中。

1995 — 命名：Jakob Nielsen 将 progressive disclosure 确立为交互设计模式的正式名称，应用于应用程序和网站设计。

2006 — 系统化：Nielsen 在 Nielsen Norman Group 发表权威文章，将渐进呈现定义为"先向用户展示少量最重要的选项，仅在用户请求时提供更多专特选项"[1]，并明确区分了 Progressive Disclosure 与 Staged Disclosure 两种变体。

2010s–2020s — 扩展应用：随着移动端和复杂 SaaS 应用的兴起，渐进呈现从桌面应用扩展到响应式 Web、移动 App、AI 界面和内容设计领域。

渐进呈现的运作机制可以归纳为两条规则[1]：

  1. 初始显示核心选项 — 只展示用户最频繁使用的功能或最重要的信息。
  2. 按需揭示细节 — 通过明确的交互入口（按钮、链接、展开控件）让用户在需要时获取更多内容。

经典案例是 macOS 的打印对话框：初始界面只显示份数和打印机选择；点击"显示详细信息"后才展开纸张大小、缩放、双面打印等高级设置。

渐进呈现改善可用性的三个维度[1]：

  • 可学习性（Learnability）— 新手用户聚焦于核心功能，降低学习门槛。
  • 使用效率（Efficiency）— 所有用户都不需要扫描大量不常用功能。
  • 错误率（Error Rate）— 隐藏高级设置防止新手误操作。

Nielsen 的研究表明，帮助用户优先关注核心功能不会导致其对系统产生"受限的心智模型" — 相反，用户在分层引导下对系统的理解更准确[1]。

分类 机制 典型场景 UI 载体
Step-by-step（步骤式） 将复杂任务拆分为顺序步骤，每步只展示对应信息 电商结账流程、注册向导、配置表单 Wizard / 多步表单 / Stepper
Conditional（条件式） 隐藏特定元素和功能直到用户主动请求 高级搜索选项、打印对话框、设置面板 按钮 / 展开控件 / 下拉菜单

三种分类并非互斥 — 一个设计方案中通常会混合使用。例如电商结账（步骤式）中的运费计算提示（上下文式）和高级配送选项（条件式）[2]。

03 / Multi-Layer Perspective

超越控件：渐进呈现的五个作用层次

将渐进呈现等同于手风琴或下拉菜单，就像将"引力"等同于"苹果落地" — 只看到了最表层的现象。渐进呈现是一种信息控制哲学，在从抽象原理到具体控件的五个层次上同时运作。

渐进呈现的哲学内核不是"简化"，而是复杂性的时间维度管理。它的根本假设是：

  • 人在任何时刻能有效处理的信息量是有上限的（Miller's Law、Cognitive Load Theory）
  • 但人对复杂性的总需求并不应该被削减 — 高级用户需要高级功能，专家需要完整信息
  • 因此，问题不是"要不要呈现复杂性"，而是"在什么时刻呈现什么层次的复杂性"

这与 Don Norman 在《设计心理学》中的区分一致：复杂（complex）不等于复杂难用（complicated）。一架飞机的驾驶舱是复杂的，但经过训练的飞行员不会觉得它复杂难用 — 因为飞行训练本身就是渐进呈现的过程：先学直飞，再学转弯，再学紧急处置。

这一层的核心洞察是：渐进呈现不是在做减法，而是在做排序 — 把所有信息按"用户此刻需要的概率"排列，高概率者先出场。

在信息架构层面，渐进呈现决定了内容的层级深度、优先级排序和导航路径。它回答的核心问题是：

  • 什么在首层？ — 用户最频繁需要的内容，不需要任何交互即可获取
  • 什么在次层？ — 有需求但频率较低的内容，通过一次明确交互可达
  • 什么可以省略？ — 只有极少数用户在极少数情况下需要的内容

超文本（Hypertext）本身就是渐进呈现在信息架构中的天然实现：高层页面提供概述和简化描述，低层页面填充细节[1]。维基百科的每一篇文章都遵循这个模式 — 首段是摘要，正文逐步展开，参考文献在末尾。这不是UI 设计决策，而是信息架构决策。

Nielsen 提出的两条信息架构准则至今适用[1]：(1) 首层和次层的功能划分必须正确 — 首层不能有让用户困惑的复杂功能，次层不能遗漏高频功能；(2) 层级间的过渡入口必须有强"信息线索"（information scent） — 用户通过入口标签即可预判下一层的内容。

在产品策略层面，渐进呈现是用户生命周期管理的核心模式。产品经理可能不会使用"progressive disclosure"这个术语，但以下策略本质上都是渐进呈现：

  • 新手引导（Onboarding） — Slack 的 Slackbot 引导用户先学发消息，再逐步介绍频道、线程、集成。这不是 UI 模式选择，而是战略性地控制用户在生命周期早期接触的复杂性[7]。
  • 功能解锁（Feature Gating） — 游戏设计中的"关卡解锁"是最纯粹的 progressive enabling：玩家必须掌握当前复杂度后才能接触下一层。Figma、Notion 等工具也采用类似策略，新用户看到的是简化界面，使用深度增加后才浮现高级功能。
  • 定价阶梯（Pricing Tiers） — Free → Pro → Enterprise 的 SaaS 定价模型，本质上是对功能复杂性的商业化渐进呈现：用户在 Free 层验证核心价值，产生深度需求后才接触（并愿意为之付费的）高级功能。
  • API 版本策略 — 开发者工具通常提供 Quick Start（核心 API, 3 个调用完成任务）→ Advanced Guide（完整参数）→ API Reference（每个字段）的三级文档结构。

产品策略层的渐进呈现与 UI 层的关键区别：它的"展开"触发器不是按钮点击，而是用户行为和时间 — 用户的使用深度、付费决策、技能成长决定了何时"披露"下一层。

渐进呈现在叙事和教育领域有着更古老、更根本的传统 — 甚至早于任何数字界面的存在：

  • 文学中的悬念结构 — 推理小说不会在第一页公布凶手。每一章揭示一条线索，读者的认知负荷始终被控制在"足以保持兴趣但不会剧透"的区间。悬念本质上就是对叙事信息的渐进呈现。
  • 教育中的脚手架理论 — Vygotsky 的"最近发展区"(ZPD) 概念指出：学习者只能理解略超于当前水平的知识。教师的任务是搭建"脚手架"(scaffolding)，让学生逐步攀升。这与渐进呈现的核心逻辑完全一致：先巩固基础，按需升级。
  • 新闻中的倒金字塔 — 将最重要的信息（who/what/when/where）放在导语，细节按重要性递减排列。读者可以在任何位置停止阅读，已获取的信息始终是最高优先级的。这是新闻写作对渐进呈现的制度化实践。
  • 学术论文的 Abstract → Introduction → Method → Results — 读者可以只读摘要判断相关性，选择性深入方法和结果。

这些案例说明：渐进呈现不是数字时代的发明，而是人类处理信息复杂性的本能模式。数字界面只是让这种模式获得了更精确的控制手段。

渐进呈现在物理世界中同样运作，只是载体从像素变成了空间：

  • 主题公园的排队设计 — 迪士尼主题公园将排队路线设计为弯曲的、被建筑遮挡的通道。在任何位置，游客只能看到前方一小段队列。这将"90 分钟排队"的心理压力渐进呈现为"每次只看到 5 分钟"的可承受片段[3]。
  • 博物馆的展览动线 — 从入口的主题概述（首层），到各展厅的分主题（次层），再到展品边的详细说明牌（三层）。参观者可以在任何层次获得完整但深度不同的体验。
  • 城市的路标系统 — 高速公路标牌只显示方向和出口编号（首层）；出口匝道标牌增加目的地名称（次层）；到达后的本地指示牌提供具体地址（三层）。每个决策点只呈现当前决策所需的最少信息。
  • 安藤忠雄的建筑序列 — 光之教堂通过狭长走廊压缩视野，然后突然打开十字形光缝 — 用空间的渐进呈现制造情感冲击。建筑师理解：同时给出全部体验等于没有体验。

物理世界的案例证明渐进呈现是一个超越介质的设计原理。无论载体是屏幕、纸张、空间还是时间，核心逻辑不变：按需求概率和认知容量控制信息流的密度和时序。

04 / UI Implementation Patterns

六种 UI 实现模式

渐进呈现在界面层面有六种成熟的实现模式，每种适用于不同的信息结构和交互场景。

📂

手风琴（Accordion）

用户控制展开/折叠，适合 FAQ、产品详情、文档章节等需要选择性深入的场景。每个手风琴项是一个独立的信息块，用户可以只阅读需要的部分[2]。

优势：减少滚动、支持非线性浏览。
风险：内容过多时可能需要用户逐个展开才能找到信息。

📑

标签页（Tabs）

将内容按类别组织到平行标签中，用户在标签间切换以查看不同维度的信息。适合内容量相近、维度明确的场景（如产品规格/评价/Q&A）[2]。

优势：分类清晰、减少滚动。
风险：标签过多或标签命名不清会增加认知负担。

📋

下拉菜单（Dropdown）

隐藏长列表以保持界面整洁。适合国家/城市选择、排序筛选等选项多但同一时刻只需一个值的场景[2]。

优势：空间效率极高。
风险：长列表内的选项如果没有搜索/分组，扫描成本可能很高。

💬

对话框与工具提示（Dialog / Tooltip）

在用户 hover 或 click 时提供额外信息，不离开当前任务。macOS 打印对话框的"显示详细信息"、表单字段的帮助图标都属于此类[1][2]。

优势：上下文相关、不打断主流程。
风险：触发方式不明显时用户可能完全忽略。

📜

滚动呈现（Scroll Reveal）

将最重要的内容放在可视区域上方（above the fold），补充信息通过滚动获取。落地页、长文章常用此模式 — 用户可以立即决策或滚动了解更多[2]。

优势：自然的阅读流、无需额外交互学习。
风险：重要信息如果在 fold 以下可能被遗漏。

🧙

向导（Wizard / Stepper）

将复杂流程拆分为线性步骤，每步只显示当前需要的信息。电商结账、账户设置、配置流程的标准模式[1]。

优势：每步简单明确、支持中途保存。
风险：步骤间相互依赖时用户需要频繁返回修改。

场景 推荐模式 原因
多步骤线性任务（结账、注册） Wizard / Stepper 步骤间有依赖关系，线性流程降低出错概率
高级设置 / 可选功能 Dialog / 展开按钮 大多数用户不需要，按需显示即可
内容密集型页面（文档、FAQ） Accordion 用户可选择性深入感兴趣的主题
同级维度切换（规格/评价/Q&A） Tabs 内容量相近，用户需要在维度间比较
长列表选择（国家、分类） Dropdown + 搜索 高空间效率，搜索解决扫描成本问题
长文章 / 落地页 Scroll Reveal 自然阅读流，无额外学习成本
字段级帮助 / 术语解释 Tooltip / Popover 不离开当前上下文即可获取信息

05 / Design Process

五步设计流程

渐进呈现不是在完成设计后"把一些东西藏起来"，而是一个从用户研究开始的系统化设计过程[2]。

1

识别用户需求

通过用户访谈、可用性测试和使用频率统计，确定用户最常使用的功能和最急需的信息。GOV.UK 银行假日页面的重新设计就是从用户研究中发现"大多数人只想知道下一个假日的日期"这一关键洞察开始的[2]。对于已有产品，代码埋点的功能使用频率数据是最可靠的依据[1]。

2

优先级排序

使用卡片分类（card sorting）和亲和图（affinity mapping）将功能和信息分为核心层（首屏）和次要层（按需显示）。核心层的选择标准：高频使用、任务关键、用户期望[1]。需要注意的是：核心层不能包含会让用户困惑的功能，否则反而降低效率。

3

确定详情层级

为每个信息块确定展示的详细程度。首屏通常只需要标题级摘要或关键指标；展开后提供完整解释。实践中不要超过 2 层层级 — 3 层以上的设计通常导致用户迷失[1]。如果功能多到需要 3 层以上，应优先考虑简化整体设计。

4

设计过渡入口

从首层到次层的过渡必须满足两个条件[1]：(1) 机制简单 — 交互方式直观（一次点击/展开）；(2) 信息线索清晰 — 按钮或链接的标签必须准确传达次层内容的预期（即具备强"信息气味"）。标签模糊的"更多"或"高级"按钮会降低可发现性。

5

原型与测试

构建交互原型进行可用性测试[2]，重点验证两点：(1) 用户能否在核心层完成主要任务？(2) 当用户需要次级功能时，能否顺利找到过渡入口？测试中常见的问题是：过度隐藏导致用户找不到需要的功能。

06 / Case Studies

实战案例分析

以下四个案例展示了渐进呈现在不同领域的应用方式，以及各自的设计权衡。

Government

GOV.UK 银行假日页面

原始设计密集展示所有假日信息，用户反馈"信息过载、找不到想要的"。用户研究发现核心需求是"下一个假日是哪天"。

  • 重新设计：最近的假日放在页面顶部，使用醒目的视觉层级；后续假日按时间顺序以较小字体列出
  • 模式：Scroll Reveal + 视觉层级
  • 效果：这个案例成为英国政府数字服务（GDS）最经典的内容设计成功案例[2]
Search

Google 搜索 → 高级搜索

Google 首页只有一个搜索框和两个按钮 — 极致的渐进呈现[2]。

  • 首层：单一输入框，满足 99% 用户的搜索需求
  • 次层：高级搜索页提供语言、地区、精确短语、更新日期等多字段筛选
  • 设计亮点：首层的简洁性使 Google 搜索成为互联网上最易学的界面之一
OS Interface

macOS 打印对话框

渐进呈现的经典教科书案例[1]，也是 Nielsen 在 2006 年原文中重点讨论的例子。

  • 首层：打印机选择、份数、页面范围
  • 次层："显示详细信息"按钮展开纸张大小、缩放、双面打印、对齐等高级选项
  • 设计权衡：Nielsen 指出近年来打印对话框趋于"膨胀"，部分应用在首层展示了过多选项[1]
Knowledge Platform

Knowledge Hub 文章系统

本站采用了多层渐进呈现设计来展示技术研究内容。

  • 首层：Hero 区域展示核心观点和关键指标，用户可以在 10 秒内判断文章是否相关
  • 次层：各 Section 通过 scroll-reveal 动画渐次呈现，手风琴组件允许用户选择性深入
  • 模式组合：Scroll Reveal（整体结构）+ Accordion（深度内容）+ Tooltip（术语解释），三种模式协同工作

07 / Anti-Patterns & Boundaries

反模式与设计边界

渐进呈现并非万能药。错误的应用会产生比不分层更糟的体验。以下是四种常见反模式和该模式的适用边界。

反模式 1：过度隐藏（Over-Hiding）

将用户常用功能错误地归入次级层。如果用户频繁需要展开才能完成常规任务，渐进呈现反而增加了操作步骤。

诊断方法：如果用户测试中超过 30% 的参与者需要进入次级层才能完成核心任务，说明首级/次级的划分有误。

反模式 2：层级过深（Deep Nesting）

超过 2 层的嵌套渐进呈现导致用户迷失方向。Nielsen 明确指出："在实践中，超过 2 层的设计通常可用性很差，因为用户经常在层级间迷失"[1]。

替代方案：如果功能多到需要 3 层以上，优先简化功能集；不能简化时，将高级功能按逻辑分组为多个平行的二级入口。

反模式 3：信息线索缺失（Missing Scent）

过渡入口的标签含糊不清（如只写"更多"、"高级"或一个 ··· 图标），用户无法预判展开后的内容。

改进方法：按钮标签应包含具体内容预期，如"显示详细打印设置"比"高级"提供更强的信息线索[1]。

反模式 4：步骤间强依赖的向导（Interdependent Wizards）

当向导的各步骤需要频繁交叉修改时，线性流程反而阻碍了探索。Nielsen 引用的酒店预订案例表明：房型、价格、日期之间有密集的 what-if 交互，强制分页增加了认知成本[1]。

替代方案：将相互依赖的信息放在同一页面，仅将独立步骤（如支付信息）分离到次级页面。

渐进呈现并非所有场景都适用。以下检查表帮助判断是否应该使用：

条件 适合渐进呈现 不适合渐进呈现
功能使用频率差异 核心功能与高级功能频率差异大 大部分功能使用频率相近
用户群体差异 新手与专家共用同一界面 所有用户都是同等水平的专家
信息量 内容/功能数量超出单屏承载能力 信息量本身就很少

响应时间要求 用户需要快速做出首次决策 用户需要全面比较后再决策

08 / AI-Collaborative Design

AI 协同应用中的渐进呈现

AI 能力的引入使渐进呈现从静态分层演变为动态自适应。但 AI 不仅是渐进呈现的"增强工具"— 它同时创造了新的复杂性层，也需要被渐进呈现来管理。以下从架构模式和具体案例两个维度分析。

传统渐进呈现的分层依据是使用频率（高频功能首层，低频功能次层）。AI 协同应用引入了一个新的分层维度：模型置信度。

  • GitHub Copilot 的三层置信度模型 — 高置信度建议以灰色内联文本直接显示在光标后（Inline Suggestion，零交互成本）；中置信度建议需要用户通过 Tab 键选择显示多个候选（Next Edit Suggestions）；低置信度或复杂任务需要用户主动打开 Chat 或 Agent Mode 对话。这是一个经典的渐进呈现架构：AI 按自身确定性自动选择披露层级。
  • Google Search 的 AI Overview — 搜索结果页将 AI 生成的摘要放在首层（高置信度综合答案），来源链接放在次层（供验证和深入），完整搜索结果保留在第三层。用户可以在摘要层获得即时答案，选择性深入来源。
  • Gmail Smart Reply / Smart Compose — 短回复建议以 3 个芯片按钮呈现在首层（一次点击完成），完整编辑在次层（用户产生更复杂需求时切换到全量编辑器）。AI 根据邮件内容的置信度决定是否显示建议 — 当判断不适合自动回复时，首层完全不呈现 AI 功能。

架构要点：置信度驱动的分层使渐进呈现从"设计时静态配置"变为"运行时动态决策"。但必须保留用户的覆盖路径— 即使 AI 选择不在首层呈现，用户仍应有明确方式手动触发 AI 功能。

AI 模型生成的内容天然具有多层结构。如何呈现这些层级，决定了用户的认知效率和信任水平：

  • ChatGPT / Claude 的可折叠推理过程 — Claude 的 extended thinking 模式将推理链（chain-of-thought）折叠在收起区域，先呈现最终结论。用户可以展开查看推理步骤以建立信任或发现错误。这是标准的条件式渐进呈现— 但触发机制从"点击按钮"变为"点击展开推理链"。
  • Perplexity 的答案 + 引用 + 来源 — 首层是直接答案，行内引用标注 [1][2][3] 是次层入口，点击引用展开完整来源段落是第三层。这个三层结构让用户在几秒内获得答案，同时保持了可溯源性。
  • Microsoft Copilot in Office 的生成 → 审阅 → 修改 — Word/PPT 中的 Copilot 先生成完整草稿（首层），底部提供"保留/修改/丢弃"三个选择（次层决策入口），只有用户选择"修改"时才展开详细编辑界面。这避免了"AI 生成后用户不知道下一步做什么"的认知瘫痪。

架构要点：AI 输出应遵循"先结论后理由"（conclusion-first）的渐进呈现原则 — 与学术论文的 Abstract → Method → Results 结构一致。推理过程是"可选的详情层"，不是必须经历的路径。Google PAIR Guidebook 将此归纳为"帮助用户校准对 AI 的信任 — 提供恰到好处的解释，既不过载也不省略"[10]。

AI 协同应用面临一个传统软件不存在的渐进呈现问题：自主权的渐进披露 — 用户愿意让 AI 做多少事？

  • GitHub Copilot 的三级委托模型 — Code Completion（AI 补全一行，用户逐行审核）→ Agent Mode（AI 规划多步任务，用户在 IDE 内审核每步）→ Cloud Agent（AI 异步执行完整任务，提 PR 后用户审核）。这三个层级的本质是渐进披露 AI 的自主权 — 从"AI 辅助我"到"AI 替我执行"。
  • Figma AI 的建议 → 生成 → 自主模型 — Figma AI 从 Auto Layout 建议（首层，用户控制）到 Design Generation（次层，AI 生成用户审核）到即将推出的整页自动排版。设计师可以选择在任何层级停下来手动接管。
  • Tesla Autopilot 的 L2 → FSD 进阶 — 自适应巡航（首层）→ 自动变道（次层）→ 城市自动驾驶（三层），每层都需要用户先在前一层积累信任和经验。这不是 UI 模式，而是对"人机控制权"的渐进披露。

架构要点：Google PAIR Guidebook 将此归纳为"为适当级别的用户自主权而设计" — AI 产品应根据任务类型、用户专业度和所需纠偏成本，设计不同级别的自主权选项[10]。渐进呈现的原则在此扩展为：先给用户低自主权（高控制感）选项，仅在用户建立信任后才呈现高自主权（低控制感）选项。

传统渐进呈现是静态分层— 首层、次层在设计时固定。AI 使渐进呈现进化为动态自适应：

  • Notion AI 的内联 vs 全页模式 — 在文本中输入 "/" 触发 AI 内联补全（首层，低干扰），单独页面的 AI 写作助手提供完整指令面板（次层，高功能密度）。首层适合轻量编辑，次层适合深度创作 — AI 根据用户的输入长度和上下文自动调整建议的详细程度。
  • Adobe Firefly 的简单 → 高级提示 — 首层只需一行文字描述即可生成图片；点击"高级设置"展开风格参考、宽高比、负面提示等参数。但 AI 会基于用户过去的生成历史，自动预填充部分高级参数 — 将静态的"显示/隐藏"升级为"基于行为模型的智能预填"。
  • Spotify 的个性化首页 — 首页不是所有用户看到相同的静态布局，而是 AI 基于听歌历史动态重排 "Daily Mix"、"Discover Weekly"、"Release Radar" 等模块的顺序和内容。这是渐进呈现的"第三种触发器" — 不是用户点击，不是系统预设，而是行为模型驱动的动态首层。

架构要点：自适应界面的风险是可预测性下降。Nielsen 的信息线索原则在 AI 场景中演化为：用户必须始终能理解"为什么 AI 在此刻向我展示这些内容"。如果 AI 的分层逻辑不可解释，用户的信任会快速消耗 — 即使 AI 的排序是正确的[10]。

AI 协同应用在实践渐进呈现时，存在三个传统设计不会遇到的特有陷阱：

陷阱 1：幻觉被渐进呈现放大

当 AI 的错误答案被放在"可信的首层"，而纠正信息（来源、推理过程）被推到"用户通常不会点击的次层"时，渐进呈现的分层结构会系统性放大幻觉的危害。如果用户习惯了"首层答案通常正确"的心智模型，验证动作的触发频率会随信任增长而下降。

对策：高风险场景（医疗、法律、财务）应将来源验证从"按需展开"提升到"始终可见" — 牺牲首层的简洁性来保障准确性。

陷阱 2：自主权渐进导致失控

用户在低自主权层（如代码补全）建立的信任，不应被无条件迁移到高自主权层（如 AI 自动提交 PR）。每一层自主权的解锁都应有独立的信任校准过程 — 否则用户可能在没有充分理解风险的情况下"升级"AI 的决策权限。

对策：高自主权层级的首次使用应有独立的"试用 → 确认"流程，而非默认解锁。

陷阱 3：过度个性化消除探索空间

AI 过度优化"用户最可能需要的首层"时，会产生"信息茧房" — 用户永远只看到 AI 认为他们需要的内容，错过可能有价值的新发现。Spotify 的 Discover Weekly 之所以成功，是因为它刻意在个性化推荐中注入 20% 的探索性内容。

对策：始终保留"查看全部"的确定性入口，并在 AI 驱动的首层中有意掺入低概率但高价值的内容。
