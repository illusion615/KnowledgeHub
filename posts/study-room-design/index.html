Knowledge Hub：交互式知识图谱的架构与实现

Knowledge Hub 是一个纯静态知识管理站点，以全屏交互式知识图谱为导航中枢，以结构化渐进披露文章为内容载体。按 2026-04-20 最近一次提交快照，项目已经演进到 Galaxy、Cards、Infographic、Cell 四种首页布局，并补齐统一搜索、主题/语言/纹理偏好持久化、文章演示模式、自动讲解、屏幕录制、邮件分享与跨端触控交互。

01 / The Knowledge Dilemma

1. 知识管理的普遍困境

我们正处于一个信息过载与知识稀缺并存的时代。获取信息的成本趋近于零，但将信息转化为可理解、可传递、可复用的知识，依然是一项高成本活动。

01

认知负荷过重

Sweller 的认知负荷理论（1988）指出，人类工作记忆容量有限——Miller（1956）将其量化为 7±2 个信息块。一篇技术论文包含的概念远超这个阈值，大量信息被遗忘而非被理解。传统的线性阅读模式迫使读者承受全部认知负荷，无法按需分配注意力。

02

知识碎片化

对一个技术主题的完整理解往往散落在论文、官方文档、GitHub 仓库、技术博客、基准测试报告等多个来源中。Vannevar Bush 早在 1945 年的论文《As We May Think》中就预见了这一困境，并提出了通过关联路径（associative trails）连接知识的 Memex 构想。80 年后的今天，我们仍在多个浏览器标签之间反复跳转。

03

隐性知识难以外化

Nonaka 与 Takeuchi（1995）在《创造知识的企业》中提出了 SECI 模型，指出知识管理的关键挑战在于将个人的隐性知识（tacit knowledge）转化为可共享的显性知识（explicit knowledge）。传统的笔记工具擅长记录，但不擅长将碎片化的理解重组为结构化的、他人可消费的知识产品。

04

消费与输出割裂

Shneiderman（1996）提出“先总览，再缩放过滤，最后按需查看细节”的信息可视化准则，至今仍是设计黄金法则。然而多数知识工具只解决了“记录”这一环——要分享需另做 PPT，要发布需另建博客，同一份知识被反复搬运和重新格式化。

02 / Design Goals

2. 设计目标

论文、技术文档、多源调研报告——这些稠密知识的消化成本极高。Knowledge Hub 的核心目标是降低复杂知识的理解门槛：将需要数小时阅读的材料转化为可在数分钟内把握要点的交互式知识产品。

01

降低稠密知识的消化难度

一篇论文往往需要反复精读数小时才能抓住核心。Knowledge Hub 通过 Hero 指标总览、编号章节、折叠深潜区的三层渐进披露结构，让读者在 3 秒内判断价值、3 分钟内掌握框架、按需深入细节——将稠密知识的消化路径从"通读全文"变为"按需探索"。

02

多源知识的结构化整合

一个技术主题的完整理解往往散落在论文、官方文档、GitHub 仓库、基准测试等多个来源中。Knowledge Hub 将这些碎片整合进一个统一的交互式页面，用洞察卡片、对比网格、流程图等设计模式组织信息，让读者无需在十个浏览器标签之间跳转。

03

打通研究到发布的完整闭环

Knowledge Hub 的价值不只在展示，而在于把同一份研究成果贯穿为一条连续流水线：研究目标进入后，最终产出可以同时成为结构化文章、首页知识图谱节点、演示稿和可发布页面。研究、写作、注册、导出和发布不再是五套割裂工具，而是围绕一份 HTML 主源协同运行。

04

单源多输出带来的效率增益

Knowledge Hub 把“研究成果整理成网页文章”“为分享重做 PPT”“同步首页目录与元数据”三段工作收束到一个源文件和一个注册点。只要研究材料足够完整，后处理阶段的工作就从多处重复排版变成一次结构化整理 + 自动复用，从而把效率提升建立在流程复用而不是单次爆发上。

05

知识关系的可视化导航

用交互式知识图谱取代传统目录列表，让知识点之间的层级关系、主题归属和跨域关联直观可见。读者一眼看到整个知识版图的全貌，发现意想不到的知识连接，沿着关联线自然延伸学习路径。

06

长期可维护

单数据源架构（knowledge-data.js）使知识树的增删改只需编辑一个文件；共享样式与通用交互抽到 article.css、article-common.js、article-presentation.js，录屏、邮件分享和讲解设置再拆成懒加载模块。复杂能力增长，但页面主源仍保持轻量。

07

零构建部署

不依赖任何构建工具、框架或包管理器。编辑 HTML 文件后直接 git push 即完成部署。技术的简单是为了项目的轻量化——消除一切与知识创建无关的工程摩擦。

03 / Knowledge Lifecycle

3. 知识流转全生命周期

从一个研究目标到一篇可分享、可定位、可导出的知识文章，Knowledge Hub 把流程拆解为六个阶段。核心设计是让审查和注册作为独立门禁存在，而不是混入写作过程——研究成果必须经过显式检验才能进入知识树，成为可追踪的资产。

  • 1
    目标定义 — 明确研究问题、受众与输出

    输入是一个主题，但第一步不直接写文章，而是先把问题边界说清：要回答什么决策问题、面向谁、最终输出是选型文章、设计文档还是周期性雷达。只有问题边界明确，后续信息收集才不会失焦。

  • 2
    证据采集 — 收集来源锚点与测量条件

    工具：浏览器 + AI 对话 + 文档阅读。围绕已定义的问题收集论文、官方文档、模型卡、基准测试和项目源码，并同步记录量化声明的测量条件。这里的目标不是“收更多链接”，而是形成一组足以支撑文章结构的证据锚点。

  • 3
    结构化撰写 — 把证据转成单源 HTML 成品

    工具：VS Code + GitHub Copilot + article.css 组件库。此阶段把证据锚点转成 Hero 指标、章节主线、折叠深潜区、对比卡片和参考文献。核心原则是“一次结构化，后续多处复用”：文章不是笔记草稿，而是后续首页、分享和导出的唯一主源。

  • 4
    内容审查 — 事实、逻辑、术语三条线并行审核

    文章注册前必须通过四项检查：数字前后是否一致、量化声明是否附带测量条件、术语是否先定义后使用、章节因果链是否完整闭合。审查独立于撰写环节运行，充当注册前的硬门禁——通不过审查的文章不允许写入 knowledge-data.js。

  • 5
    注册编目 — 让文章进入知识树而不是停留为孤页

    工具：knowledge-data.js。只有在文章标题、摘要、标签、父主题和关联线全部确定后，才向 knowledgeTree 写入注册条目。注册动作是原子性的——一次编辑让首页四种布局、全局搜索、标签筛选和关联线一起生效。未注册的页面即“孤页”，它对首页不可见、对搜索不可达、对关联不可用。

  • 6
    发布复用 — 用同一份 HTML 完成分享、演示与上线

    工具：article-presentation.js 及其按需懒加载的讲解、录屏、邮件分享模块 + GitHub Pages。现在一篇文章既可作为深度阅读页，也能切换为演示模式、生成自动讲解、录制讲解视频、导出 PPTX，或打开邮件分享预览。这里的价值不只是“发布更快”，而是把阅读、演示、分享和上线继续收束到同一份 HTML 主源。

已自动化的环节

六个阶段中，第 3–6 步已由工具链直接支撑：article.css 组件库驱动结构化撰写、4.3 节的审核策略驱动内容审查、knowledge-data.js 的单数据源驱动注册编目、article-presentation.js + GitHub Pages 驱动发布复用。这些环节的执行不依赖人的记忆，而是嵌入在工具行为中。

需要人工判断的环节

第 1–2 步——目标定义与证据采集——仍然主要依赖人工判断：选什么问题、面向谁、收集哪些证据锚点。这两个阶段的质量直接决定后续四步的输出上限。项目通过 Copilot 指令编码了证据采集的最低要求（测量条件、来源锚点），但研究方向本身的取舍是人工决策。

04 / Quality Assurance

4. 知识生产的质量保障

将日常知识学习转化为可重复的高质量输出，需要在研究、撰写和审核三个环节建立标准化的策略。这些策略确保每篇知识文章不是一次性产物，而是经过系统化流程验证的知识资产。

01

多源交叉验证

每个核心事实至少从两个独立来源确认：官方文档 + 论文原文、基准测试数据 + 实际复现。避免二手转述导致的信息失真。AI 辅助扩展搜索范围，但最终事实锚点必须可溯源。

02

先框架后细节

研究阶段先用 AI 对话梳理主题的宏观结构和关键维度，形成信息架构骨架。再沿着骨架逐项深入，填充数据和论据。这个顺序确保不会在细节中迷失方向，也为后续文章的章节编排奠定基础。

03

量化优先

优先收集可量化的数据点：模型参数量、基准测试分数、延迟数字、内存占用、成本估算。每个量化声明必须附带测量条件（硬件、配置、步骤）。无法量化的差异用"无数据"显式标注，而非用模糊措辞掩盖。

01

AI 生成 + 人工架构

代码层面由 GitHub Copilot 全程辅助生成 HTML/CSS/JS，人工把控的是信息架构：哪些内容放 Hero 指标、哪些归入编号章节、哪些沉入折叠深潜区。AI 负责执行，人工负责决策——每篇文章的结构设计和内容分层是核心人工工作。

02

3 秒 / 3 分钟 / 30 分钟法则

每篇文章必须服务三种阅读深度：3 秒看 Hero 指标判断价值、3 分钟扫完编号章节掌握框架、30 分钟展开折叠区深入所有细节。这个分层标准决定了每一段内容应该放在哪个层级。

03

组件化写作

内容通过标准化的设计组件表达：洞察卡片承载关键观点、对比网格呈现方案差异、流程列表展示步骤、比较表格量化对比。组件选择不是装饰性的，而是语义化的——每种组件对应一种特定的信息传递模式。

01

事实校验审核

每篇文章发布前进行事实锚点审核：检查量化数据是否附带测量条件、引用来源是否可追溯、前后数字是否一致（如 Hero 指标与正文数据）。禁止空泛修辞（"显著提升"、"最强的"），必须换成具体数字或比率。

02

信息架构审核

审核每篇文章的分层是否合理：Hero 是否只有 3-4 个最关键指标？每个章节是否有明确的单一主题？折叠区是否只包含需要深入的补充细节而非核心论点？遵循 Miller 7±2 法则，每层信息块数控制在人类工作记忆容量之内。

03

跨模型交叉审核

完成初稿后交由不同的 AI 模型进行独立审核，检查逻辑一致性、缺失前提、未定义术语和因果链断裂。不同模型的审核盲区不同，交叉审核能捕获单一模型可能忽视的问题。最终由人工判断哪些审核意见值得采纳。

01

先完成页面，再注册知识树

项目级指令约定：文章页必须已存在、标题/摘要/标签/父主题必须已定稿，才允许写入 knowledge-data.js。这条规则同时防止两种漂移——首页出现“空壳卡片”（页面未完成），或文章沦为“孤页”（页面写好但首页找不到）。

02

量化声明必须带条件

量化声明必须同步记录硬件、配置、步骤和统计口径。裸数字——即不附带条件的数字——不允许出现在 Hero 或正文中。若数据缺失，正文必须显式标注缺口位置，而不是用“显著”“约”“接近”等形容词模糊带过。

03

发布时同步正文与元数据

文章正文、knowledge-data.js 摘要、首页分组和项目指标必须在同一次提交中对齐。任何涉及文章内容变更的修改，都不允许只改一侧——正文数字更新了，摘要和指标必须同步跟进，反之亦然。

05 / Design Philosophy

5. 设计理念

Knowledge Hub 的设计理念可归结为五条原则，它们贯穿从首页知识图谱到文章演示、讲解与分享模块的所有设计决策。

01

内容即产品

每篇文章是一个独立的交互式产品页，不是博客帖子。Hero 区域的关键指标、卡片网格、对比矩阵、折叠深潜区——这些设计模式让知识像产品一样被结构化呈现和消费。

02

约束驱动创新

刻意排除框架和构建工具，迫使所有交互效果必须在原生 Web 标准内实现。这个约束催生了纯 CSS 变量驱动的暗色模式、IntersectionObserver 滚动动画、HTML data 属性驱动的双语切换等方案。

03

单数据源真相

知识图谱的所有节点、层级和关联关系集中在 knowledge-data.js 的一个扁平数组中（邻接表结构）。首页四种视图模式、卡片过滤、标签云都从这一个数据源自动派生，消除数据不一致的可能。

04

多维度探索

同一份知识数据通过四种空间隐喻来呈现：Galaxy 用轨道系统表达主题引力，Cards 用分组卡片表达快速扫描，Infographic 用弧线轮盘表达层级深度，Cell 用有机包裹表达主题容器与内容密度。用户按认知任务选择视图，而不是被单一导航方式绑定。

05

偏好即状态

主题、语言、布局、文章打开方式、背景纹理都通过 localStorage 持久化，跨页面保持一致。文章页自动继承首页设置，首页则把偏好解释为完整的探索状态，而不只是视觉开关。无需注册、无需后端、无需数据库——浏览器就是状态存储。

06 / Design Approach

6. 设计思路

从知识结构建模到前端渲染，Knowledge Hub 的设计围绕三个核心思路展开。

知识结构采用扁平数组 + parentId 引用的邻接表模式（knowledgeTree[]）。每个节点只需声明自己的 id 和 parentId，树的深度不受限制。这种设计比嵌套 JSON 更易于增删节点——任何结构调整只需修改一个 parentId 值。

首页通过一组树工具函数（getChildren / getDepth / getRoots / getAllArticles）从扁平数组实时计算出三套渲染所需的派生数据：planets（根节点）、groups（子分组）、posts（文章）。数据永远从单源推导，不存在手动同步。

Galaxy

星轨模式

根节点渲染为大型行星，所有子主题按无限深度的父子轨道递归展开。主题尺寸和运动速度由后代文章数驱动：内容越多，节点越大、轨道越慢。支持桌面滚轮缩放、拖拽平移，以及 iPad 上的单指拖拽与双指捏合。

Cards

卡片模式

将知识节点按主题递归分组，渲染为响应式网格卡片。文章按日期从旧到新排序，整张卡片可点击打开。适合快速浏览全量内容、按搜索结果扫描匹配条目，或从某一主题直接下钻到文章。

Infographic

弧线轮盘模式

以同心弧线展开层级结构，topic 和 article 都直接渲染在轮幅上。文章轮幅包含日期、标题、标签和摘要，减少额外弹窗依赖。桌面端支持滚轮和键盘，iPad 端支持上下滑动切换条目、左右滑动进退层级。

Cell

细胞模式

以有机细胞团块表达主题边界：topic 逐层包裹 topic 与 article，article 以可点击节点停靠在主题容器内部。布局由轻量力导向模拟驱动，支持拖拽平移、双指捏合缩放，以及和 Galaxy 共用的统一摘要悬浮卡。

每篇文章遵循统一的四层结构，由浅入深引导阅读：

01

Hero 总览层

标题 + 一句话描述 + 3-4 个关键指标数字。让读者在 3 秒内判断这篇文章是否值得深入阅读。

02

编号章节层

章节以 01 / Section Name 格式编号，配 section-kicker 标签。每章有导入段落和洞察卡片网格，覆盖该主题的核心观点。

03

折叠深潜层

通过 data-accordion 标记的折叠区域承载详细论述、对比表格、代码示例。默认折叠，只有感兴趣的读者点击展开。降低认知负荷。

04

滚动揭示动画

所有章节通过 IntersectionObserver 实现滚动到视口时的淡入动画（data-reveal），配合 load-in 类的首屏错列延迟动画，创造流畅的阅读节奏。

Notion / Obsidian

强大的个人知识库工具，但核心限制在分享端：Notion 发布的页面是平台锁定的渲染视图，无法自定义交互和视觉呈现；Obsidian Publish 受限于 Markdown 渲染能力，无法实现知识图谱可视化、幻灯片模式或 PPTX 导出。两者都无法做到“一份内容同时服务阅读、演示和分享”。

Hugo / Next.js / VitePress

静态站点生成器解决了部署问题，但引入了构建工具链的维护负担（node_modules、版本升级、构建配置）。更重要的是，它们的模板系统为通用博客设计，无法原生支持每篇文章独立的交互式布局（Hero 指标、折叠深潜、卡片网格）。Knowledge Hub 的每篇文章都是一个自包含的交互产品页——这在模板引擎中反而是约束。

Knowledge Hub 选择的路径是：用原生 HTML 获得对每篇文章的完全布局控制，用共享 CSS 组件库（article.css）保持一致性，用单数据源（knowledge-data.js）驱动全站导航——在零构建的前提下兼得灵活性和可维护性。

07 / Architecture

7. 设计架构

Knowledge Hub 的当前架构可分为四层：展示层、能力编排层、共享组件层和数据层。

展示层

知识图谱首页 + 交互式文章页

首页提供 Galaxy / Cards / Infographic / Cell 四种知识图谱视图，统一挂接搜索框、纹理选择、文章打开方式和悬浮信息卡。每篇文章是独立的交互式产品页，包含 Hero 指标区、编号章节、折叠深潜区和卡片网格，并自动继承主题、语言与阅读偏好。

能力层

演示编排 · 自动讲解 · 录屏 · 分享导出

article-presentation.js 负责步骤收集、Deck 导航、分享菜单与跨模式状态切换；并按需懒加载讲解设置、屏幕录制和邮件分享模块。当前能力覆盖自动讲解、字幕叠加、Focus Mode、16:9 / 9:16 / 1:1 录制比例、PPTX 导出与 AI 问答接力。

组件层

共享样式框架 + 基础初始化 + 轻量扩展

article.css 提供 Hero、章节、折叠区、网格和暗色模式等共享样式；article-common.js 统一处理 reveal、导航高亮、accordion 状态和 localStorage 偏好同步；scrollbar.js、lightbox 与讲解 UI 为跨文章复用的轻量扩展，保证页面主源不被重复逻辑污染。

数据层

knowledge-data.js — 唯一数据源

knowledgeTree[] 以扁平邻接表存储全部层级、元数据与 relations。首页通过树工具函数实时派生 planets、groups、posts 等运行态数据，因此节点数扩张到 96 个、布局扩张到 4 种之后，仍然保持单一数据真相。改变任何节点的 parentId 即可重组知识结构。

08 / Feature Matrix

8. 功能矩阵

以下是 Knowledge Hub 的完整功能列表，按功能域分组。

01

四视图知识图谱

同一份 knowledgeTree 数据同时驱动 Galaxy、Cards、Infographic、Cell 四种布局。用户可以在空间探索、快速扫描、层级轮盘和有机细胞容器之间切换，而无需切换数据模型。

02

统一搜索与过滤

头部搜索框跨四种视图复用：Galaxy 与 Cell 对不匹配节点做弱化处理，Cards 直接筛卡，Infographic 只保留匹配主题或命中后代文章的轮幅。标签云仍然保留，用于主题式过滤。

03

桌面与 iPad 手势

Galaxy 与 Cell 支持滚轮缩放、拖拽平移，也支持 iPad 上的单指拖拽与双指捏合。Infographic 除了滚轮和键盘外，还支持上下滑动切换条目、左右滑动进退层级，补齐平板浏览器的可操作性。

04

状态化设置面板

首页设置面板集中管理布局、背景纹理、文章打开方式、语言和主题。背景纹理提供 5 种选择，文章打开支持当前页与新页面两种模式；这些偏好通过 localStorage 跨页面继承。

01

演示/幻灯片模式

按 P 或点击按钮进入全屏幻灯片模式。Hero + 各 section 自动成为步骤，含折叠深潜区的 section 会自动拆成概览页 + 子步骤，Deck 顶栏统一负责标题、页码与键盘导航。

02

自动讲解与字幕

演示模式已接入自动讲解：系统可按页面内容生成旁白脚本，配合字幕叠加、语速/语言/音色设置以及文章级 localStorage 记忆，实现“可读 + 可讲”的双模式输出。

03

聚焦模式与屏幕录制

讲解时可开启 Focus Mode，自动淡化非锚点内容；同时支持 getDisplayMedia 屏幕录制，录制比例可切换 16:9、9:16 和 1:1，并可与讲解音频联动保存。

04

邮件分享与 PPTX 导出

分享菜单已同时支持 PPTX 导出和邮件分享预览：邮件模式会展开折叠内容、移除交互控件并强制切换到易读亮色版本，减少二次整理成本。

05

AI 问答助手

浮窗式 Q&A 小部件会自动提取当前文章全文作为上下文，调用外部 LLM API（支持 Ollama / OpenAI 兼容接口）回答问题，使文章具备按需追问能力。

06

图片灯箱

点击文章中的图片即可全屏查看，Esc 或点击背景关闭，自动隐藏页面滚动条，保证图表和设计稿在深度阅读时可独立放大审视。

01

暗色/亮色主题

CSS 自定义属性驱动的完整暗色模式。:root 定义亮色变量，[data-theme="dark"] 覆写。自动检测系统偏好，localStorage 跨页面持久化。

02

中英双语切换

所有文本元素通过 data-zh / data-en 属性支持双语。JS 扫描所有带标记的元素并替换 innerHTML，语言偏好跨页面保持一致。

03

响应式布局

640px 断点适配移动端和桌面端。Hero 网格、卡片网格、导航栏在窄屏下自动堆叠为单列布局；首页交互层则进一步为 iPad Safari 提供触摸拖拽、捏合缩放和滑动导航。

10 / Coding Conventions

10. 编码约定

Knowledge Hub 项目遵循的关键编码约定与禁止模式。

✓

纯 Vanilla JS

不使用 jQuery、React 或任何 JS 框架。所有脚本使用 ES5 兼容语法（避免箭头函数）。var 声明置于作用域顶部。

✓

CSS 自定义属性主题

:root 定义亮色变量，[data-theme="dark"] 覆写。所有颜色引用 var(--xxx)，单点切换即全站暗色。

✓

DOMContentLoaded 包裹

所有页面脚本包裹在 DOMContentLoaded 事件中，确保 DOM 就绪后再执行。事件绑定使用 addEventListener，不使用内联处理器。

✓

共享样式优先

文章页先引用 assets/article.css，只在页内 <style> 中放置文章特定覆写（如 Hero 标题字号）。暗色模式变体统一放在 article.css 中。

✗

禁止框架/打包器

不引入 node_modules、webpack、vite 或任何构建工具链。部署就是 git push。

✗

禁止箭头函数

文章脚本中使用 function 表达式和 forEach，保持广泛的浏览器兼容性。

✗

禁止硬编码数据

知识图谱数据只存在于 knowledge-data.js 中，禁止在 index.html 中直接写入知识点数据。

✗

禁止 Markdown 文章

所有文章内容直接使用 HTML，不创建 Markdown 原始文件。每篇文章是一个独立的、可完全控制布局的交互式页面。
