MLX Optimization Blueprint

Apple Silicon 上的 MLX 模型优化实践

在 AI-MUD 项目中，我们从零构建了基于 Apple MLX 框架的 FLUX 图像生成管线（Z-Image-Turbo），实现了从 23GB 显存到 6GB 的4-bit量化压缩，并通过 mx.compile、分阶段内存释放和微服务隔离等策略，使 16GB Mac 也能运行 6B 参数的图像生成模型。本文记录了整个优化历程和关键经验。

管线构建路径

  1. 1
    编码器层

  5. 5
    性能优化

03 / Optimization Strategies

3. 优化策略详解

将 DiT backbone 从 bf16 量化到 4-bit，显存从 23GB 降至 6GB。关键取舍：推理精度有微小损失，但使 16GB Mac 成为可能。

同一时刻只有一个大组件在内存中。T5编码完成后释放 → 加载 DiT → DiT推理完成后释放 → 加载 VAE 解码。峰值内存下降 50%+。

Apple MLX 的 JIT 图编译，约 2 倍推理加速。需要注意首次调用的编译开销。

Apple Silicon 上 PyTorch MPS 后端启用 bfloat16，图像生成速度提升 15-20 倍。

零 ML 依赖路由层 + 隔离引擎进程在独立 venv 中运行。解决了 torch/mflux/MLX 之间的依赖冲突。

04 / LLM Client Architecture

4. LLM 客户端架构优化

Three-Layer Architecture

llm_client.py 三层架构

  • Layer 1 (Role API) — call_role_model(role, messages, stream) — 业务层唯一入口。解析 role → model → provider → endpoint。
  • Layer 2 (Dispatch) — call_model(provider, endpoint, model_id) — 解析参数、获取 provider、分派调用。
  • Provider Layer — LlmProvider Protocol + 6 个方法（call, stream, probe, warmup, heartbeat, build_auth_headers）。三个实现：OllamaProvider, OpenAICompatProvider, AzureProvider。Provider 之间零交叉依赖。
Key Optimizations

关键优化

  • 重构成果 — 1905 行 → 995 行（-48%），消除 ~220 行跨 Provider 重复代码
  • Dual Semaphore — global + per-model 并发控制
  • Per-model CircuitBreaker — 模型级熔断器
  • PER_MODEL_CONCURRENCY 2→6 — 修复 storyteller + supervisor 共用模型时的饥饿问题
  • Connection pool + httpx.Client 复用 — 单例 HTTP 客户端

Key Insights

基准测试启示

NVIDIA GPU 在图像生成上仍有绝对速度优势（5-11x），但 Apple Silicon 的统一内存允许加载远超显存限制的模型。MLX 原生管线在 Apple Silicon 上接近 MPS torch 性能，同时具有更好的内存管理和系统集成。LLM 推理场景下 Ollama 的 MLX 引擎已达到实用水平。

06 / Lessons Learned

6. 经验总结

  • 自建管线给予了完全控制权 — 权重 key 映射、内存生命周期、进度回调都成为一等公民，而非依赖黑盒框架的内部状态。
  • Apple Silicon 统一内存是双刃剑 — 适合装大模型（64GB 统一内存可装 35B 参数模型），但 MPS 计算速度显著落后 NVIDIA。
  • 进程隔离是必要的 — 不同推理引擎有冲突依赖，独立 venv + 子进程是唯一可行方案。
  • Provider 抽象回报巨大 — 新增一个 LLM Provider 只需实现一个 Protocol 文件，对分派层和业务层零改动。
  • 4-bit 量化是普及的关键 — 从 23GB 降到 6GB 峰值内存，让 16GB Mac 也能运行。
  • 最大的优化是架构级的 — 合并 storyteller + state analyst 为单次 LLM 调用，消除了每个玩家动作一次额外的模型往返。
