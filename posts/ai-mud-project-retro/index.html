AI-MUD 项目回顾：产品级 Vibe Coding 实践

AI-MUD Episode 1——"魏晋风骨：士人志"是一个由单人工作室（Boss + AI Copilot）开发的 AI 驱动交互叙事游戏。经历 156+ 轮正式迭代、14 次代码审查和 17 个架构决策，本文回顾了整个项目的历程，分享了如何在保持"vibe coding"敏捷性的同时实现产品级工程质量的经验。

01 / What Is AI-MUD

1. 什么是 AI-MUD

01

项目定义

AI-MUD Episode 1 是一款 AI 驱动的交互式叙事游戏，以魏晋时期为背景，玩家通过文字指令与 AI 叙事者互动。

02

核心机制

"风骨度"评分系统——玩家的道德选择会触发隐藏剧情线、对话分支和不同结局。

03

技术栈

Director API (FastAPI) + Image Server + Voice Server + LLM Server + Ollama + PostgreSQL + pgvector。纯原生 JS 前端，无框架。

04

产品指标

70% 首次会话完成率、≥80% 好评率、<1% 崩溃率、10-20 分钟目标会话时长。

Scale

项目规模

206 个 API 端点、22 个路由文件、80+ 后端文件、49+ 前端文件（33 JS + 14 CSS）、5 个独立服务进程。

截至 2026-04-09，仓库累计 150,642 行源码、1,331 份文档、45 次 Git 提交，首个提交日期为 2026-02-22。

Team

一人工作室

所有代码由一个人（Boss）和 AI Copilot 协作完成。没有团队分工，但有严格的角色分离——Boss 做决策和审核，AI 做分析和执行。

02 / Product-Grade Vibe Coding Protocol

2. 产品级 Vibe Coding 开发协议

每条 Copilot 回复以"报告老板"开头、"下一步："结尾。Boss 使用触发词加载对应 Playbook：新需求 / Bug / 重构 / 调研。进度优先原则——触发词输入默认为分析→记录 TODO→按序执行。

Gate-0 轮次简报 + 需求对齐 → Gate-1 设计/测试计划/文档/架构审查 → Gate-3 测试通过后才能扩展范围 → Gate-4 对齐审查 + 测试报告 → Gate-D 数据变更专项审查。

每轮迭代要求 9 份文档：REQ-ALIGN, PLAN, DESIGN, TEST-PLAN, DOC-REVIEW, ARCH-REVIEW, ARCH-DESIGN, ALIGN-REVIEW, TEST-REPORT。零延迟归档——必须在代码完成的同一会话内完成。

不允许 fallback 掩盖错误、不允许静默异常吞噬、不允许硬编码提示词、不允许绕过 RepositoryPort、不允许并行复杂特性扩展、单问题最多 3 次连续修复尝试。

04 / Code Review Practice

4. 代码审查实践

14 次代码审查（R01-R10），触发条件：每 5 轮迭代 / 阶段转换 / 单文件超 500 行（后端）或 2000 行（前端）/ Boss 要求。

Dimensions

9 维度审查

架构一致性、重复代码、命名可读性、错误处理、可测试性、性能、安全性、设计系统合规、文档健康。每个维度 ABCD 分级，任何 D 级 = 立即停止特性开发。

Impact

审查的价值

14 次审查中发现的问题推动了 llm_client.py 从 1905 行重构到 995 行（-48%）、CSS 治理体系建立、前端 ES Module 规范化、数据库访问层抽象等关键改进。

05 / Lessons Learned

5. 经验与教训

  • 门禁制度防止了 90% 的返工 — 没有 Gate-1 的设计审查，直接写代码会导致架构漂移累积到不可收拾。
  • 文档同步是最容易被跳过也最不能跳过的 — C-153 教训：7 个实体变更没有同步文档更新，后续调试成本翻倍。
  • AI Copilot 需要约束才能发挥最大价值 — 不设边界的 AI 会过度工程化、引入不必要的抽象。触发词 + Playbook 机制把 AI 限定在正确的执行模式。
  • 单人工作室最大的风险是注意力发散 — 进度优先原则和单线程工作规则是关键约束。
  • 周期性代码审查是架构健康的唯一保障 — 没有外部审查者时，结构化的自审制度是替代方案。
  • 测试不是可选的 — 464+ 测试是产品级质量的底线。每次 git commit 前必须通过测试。
  • 数据治理必须前置 — Gate-D 的诞生源于 C-153 的惨痛教训：不做数据审查就写代码 = 事后重构。
  • Vibe Coding 的"vibe"是方向感，不是随意性 — 用自然语言驱动开发是效率工具，但必须配合工程纪律。

06 / Start Your Own Vibe Coding Project

6. 如何启动你自己的产品级 Vibe Coding 项目

Checklist

启动清单

  1. 1
    定义沟通协议

    建立 Boss ↔ AI 的对话格式、触发词和 Playbook

  2. 2
    建立门禁制度

    至少 Gate-0（需求对齐）和 Gate-3（测试通过）

  3. 3
    建立归档制度

    每轮迭代必须有可追溯的决策记录

  4. 4
    设定禁止行为

    从第一天就定义 AI 的行为边界

  5. 5
    启动代码审查节奏

    每 5 轮或每阶段一次结构化审查

Maturity Signals

成熟度信号

  • 1每轮迭代有完整的 9 份归档文档
  • 2测试覆盖率持续增长
  • 3代码审查未发现 D 级问题
  • 4文档与代码同步率 100%
