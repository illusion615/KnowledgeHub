Claw Code 用户手册：从安装到精通

01 / Prerequisites

1. 环境准备

在使用 Claw Code 之前，需要完成 Rust 工具链安装和 API 认证配置。整个过程约 5 分钟。

01

Rust 工具链

需要安装 Rust 和 cargo。推荐使用 rustup 安装最新稳定版。如果你已经安装过 Rust，确认 cargo 可用即可。

02

API 认证

支持两种认证方式：设置 ANTHROPIC_API_KEY 环境变量进行直接 API 访问，或使用 claw login 命令进行 OAuth 认证。二选一即可。

03

可选配置

如需使用代理或本地服务，可设置 ANTHROPIC_BASE_URL 环境变量指向自定义 API 端点。大多数用户无需此配置。

安装步骤

# 1. 安装 Rust（如果尚未安装） $ curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh # 2. 克隆仓库 $ git clone https://github.com/illusion615/claw-code.git $ cd claw-code # 3. 构建整个工作区 $ cd rust $ cargo build --workspace # 构建完成后，CLI 二进制位于： rust/target/debug/claw # 4. 配置 API Key（二选一） $ export ANTHROPIC_API_KEY="sk-ant-..." # 或使用 OAuth 登录 $ ./target/debug/claw login

02 / Quick Start

2. 快速上手

Claw Code 提供三种基本使用模式：交互式 REPL、单次 Prompt 和 JSON 输出。30 秒即可完成第一次对话。

交互式 REPL

多轮对话模式

启动后进入交互式终端，可进行多轮对话。适合探索性开发、代码调试和需要上下文延续的任务。支持历史会话恢复。

$ cd rust $ ./target/debug/claw
单次 Prompt

一问一答模式

发送一个 prompt 并获取响应后退出。适合脚本集成、CI/CD 管道中的自动化任务和批量代码分析。支持简写形式。

# 完整形式 $ ./target/debug/claw prompt "summarize this repository" # 简写形式 $ ./target/debug/claw "explain rust/crates/runtime/src/lib.rs"

JSON 输出模式

使用 --output-format json 标志获取结构化输出，方便脚本解析和管道集成。

$ ./target/debug/claw --output-format json prompt "status" # 输出示例（结构化 JSON） { "response": "...", "usage": { "input_tokens": 1024, "output_tokens": 256 }, "model": "claude-sonnet-4-20250514" }

03 / CLI Modes

3. CLI 模式与模型控制

Claw Code 支持模型别名切换、权限模式指定、扩展思考和会话管理。这些参数可以在启动时通过命令行标志设置。

命令 / 标志 说明 示例
--model 指定模型，支持别名 opus / sonnet / haiku claw --model opus
--permission-mode 设置权限等级 claw --permission-mode read-only
--extended-thinking 启用扩展思考（链式推理） claw --extended-thinking
--session 恢复指定的历史会话 claw --session abc123
--output-format json 输出 JSON 格式，便于脚本解析 claw --output-format json prompt "..."
claw login OAuth 登录认证 claw login
claw logout 清除 OAuth 凭证 claw logout

Claw Code 内置了三个模型别名，让你无需记忆完整的模型标识符：

  • opus — 最强推理能力，适合复杂架构设计和代码审查
  • sonnet — 平衡速度与质量，日常开发的默认选择
  • haiku — 最快响应速度，适合简单查询和批量操作

每次 REPL 会话会自动保存到本地。下次启动时可通过 --session 标志恢复之前的对话上下文，包括工具调用历史和模型选择。这意味着你可以在中断后无缝继续之前的工作。

04 / Built-in Tools

4. 内置工具系统

Claw Code 暴露 40 个工具规格供 AI 代理自动调用。你无需手动执行这些工具——当你向 claw 发出指令时，它会自动选择合适的工具组合来完成任务。以下按类别介绍核心工具。

01

终端 & Shell

Bash 工具可在工作区内执行任意 Shell 命令。代理会自动用它来运行测试、安装依赖、查看 git 状态等。在 read-only 模式下，写入类命令会被拒绝。

02

文件操作

ReadFile、WriteFile、EditFile 三件套：读取文件内容（支持 offset/limit）、创建/覆写文件、精确替换文件中的文本片段。EditFile 使用精确匹配而非行号，避免并发编辑冲突。

03

搜索工具

GlobSearch 按文件名模式查找文件，GrepSearch 按正则表达式搜索文件内容。两者结合可以快速定位代码库中的任何符号、配置项或文本模式。

04

Web 工具

WebSearch 执行网络搜索获取最新信息，WebFetch 抓取指定 URL 的内容并转换为可读文本。让代理可以查阅文档、API 参考和在线资源。

05

Sub-Agent 编排

Agent 工具可以启动子代理来处理独立的子任务。主代理通过任务描述和 prompt 创建子代理，子代理完成后返回结果。适合将大任务分解为并行执行的小任务。

06

LSP 智能分析

LSP 工具连接语言服务器，支持符号查找、引用查询、定义跳转、悬浮信息和诊断信息获取。让代理理解代码语义而非仅仅搜索文本。

07

任务管理

TodoWrite 维护结构化任务列表，TaskCreate/TaskGet/TaskList 管理后台任务，TeamCreate 创建并行子代理团队，CronCreate 设置定时任务。完整的任务编排体系。

08

MCP 与 Notebook

MCP 工具连接外部 MCP 服务器扩展能力边界，NotebookEdit 直接编辑 Jupyter Notebook 的单元格。两者让 claw 的能力可以按需扩展，不受内置工具集的限制。

05 / Configuration

5. 配置体系

Claw Code 使用五级配置层级，后项覆盖前项。这种设计让你可以在全局设置默认值，再在特定仓库中覆写，最终用本地配置保存个人偏好而不影响团队。

~/.claw.json ← 优先级 1（全局） ~/.config/claw/settings.json ← 优先级 2（用户级 XDG） <repo>/.claw.json ← 优先级 3（仓库级，可提交到 git） <repo>/.claw/settings.json ← 优先级 4（项目级，可提交到 git） <repo>/.claw/settings.local.json ← 优先级 5（本地级，应加入 .gitignore）
实践建议 将团队共享的配置放在 <repo>/.claw.json 或 <repo>/.claw/settings.json 并提交到版本控制。将个人偏好（如 API Key 路径、权限模式）放在 settings.local.json 并加入 .gitignore。

在仓库根目录放置 CLAW.md 文件，Claw Code 会在每次会话启动时自动读取并注入系统 prompt。你可以在其中定义：

  • 项目架构约定和编码规范
  • 技术栈偏好（如框架、测试库选择）
  • 文件组织结构说明
  • 团队协作约定（如 commit message 格式）
  • AI 代理的行为偏好（如谨慎度、输出风格）
// .claw/settings.json — 团队共享配置 { "model": "sonnet", "permission_mode": "workspace-write", "extended_thinking": false } // .claw/settings.local.json — 个人配置 { "model": "opus", "permission_mode": "danger-full-access", "extended_thinking": true }

06 / Permissions & Security

6. 权限与安全

Claw Code 通过 PermissionEnforcer 在每次工具执行前进行守门检查。三级权限模式让你精确控制代理的操作边界，从只读审查到完全自主操作。

SAFE

read-only

只允许读取操作。所有文件写入、bash 中的写入命令均被拒绝。适合代码审查、架构分析和学习场景——让代理看但不能动。

MODERATE

workspace-write

允许在工作区边界内写入文件。任何试图跨越工作区目录的写入操作都会被阻止。适合日常开发——代理可以修改项目文件，但不会影响系统文件。

FULL

danger-full-access

完全访问权限，默认模式。代理可以执行任何操作，包括系统级命令。适合受信环境下的自主编码——代理拥有与你相同的操作系统权限。

无论权限模式如何，Claw Code 始终强制执行以下安全措施：

  • 路径穿越防护 — 检测符号链接跟随和 ../ 逃逸，阻止访问工作区外的文件
  • 大小限制 — MAX_READ_SIZE / MAX_WRITE_SIZE 约束防止意外读取或写入超大文件
  • 二进制检测 — NUL 字节检测自动阻止对二进制文件的文本操作，避免文件损坏
  • 工具调用审批 — 在非 full-access 模式下，高风险工具调用会弹出用户确认提示
  • 代码审查 / 学习新项目 → read-only
  • 日常功能开发 / Bug 修复 → workspace-write
  • 个人项目 / 受信 CI 环境 → danger-full-access
  • 团队协作项目 → 在 .claw.json 中默认 workspace-write，让个人按需覆写

07 / Slash Commands

7. Slash 命令速查

在 REPL 交互模式中，你可以使用斜杠命令来控制会话状态、切换模型和查看系统信息。这些命令不会消耗 API token。

命令 功能说明
/status 显示当前会话状态：模型、权限模式、token 用量和成本
/model 切换模型。支持完整名称或别名（opus / sonnet / haiku）
/compact 压缩会话上下文，保留关键信息并释放 token 空间
/clear 清空当前会话历史，重新开始
/permissions 查看和修改当前权限模式
/config 查看当前生效的配置项及其来源层级

08 / Advanced Usage

8. 进阶用法

掌握基础操作之后，可以解锁 Sub-Agent 编排、MCP 扩展、Hooks 集成和自动化工作流等高级能力，将 Claw Code 从交互式助手提升为自主编码系统。

1

发出指令

用自然语言描述目标，如「重构 auth 模块，提取公共接口」

2

代理分析

Claw 自动阅读相关代码、理解结构、制定计划

3

工具执行

自动调用 ReadFile、EditFile、Bash 等工具完成修改

4

验证结果

运行测试、检查编译、确认修改正确

5

报告产出

汇总修改内容，说明变更理由和验证结果

MCP（Model Context Protocol）允许 Claw Code 连接外部工具服务器，扩展内置工具集之外的能力。典型场景包括：

  • 连接数据库查询工具，让代理直接查询生产数据
  • 集成 Slack/Discord 消息推送，实现任务完成通知
  • 连接 Figma/Design 工具，让代理读取设计稿
  • 接入内部 API 网关，执行领域特定操作

MCP 生命周期管理支持部分成功——即使部分服务器连接失败，其余服务器仍可正常工作。

Hooks 允许在工具执行前后插入自定义逻辑。PreToolUse 在工具被调用前触发，可以用于：日志记录、参数校验、操作审批和安全检查。PostToolUse 在工具执行后触发，可以用于：结果转换、通知推送和指标采集。

注意 Hooks 功能当前仍在开发中（🔧 标记），接口可能在后续版本中变化。建议在非生产环境中试验。

通过 TeamCreate 和 Agent 工具，可以构建多代理并行工作流。主代理作为编排者，将任务分解并分配给多个子代理并行执行。典型场景：

  • 跨模块重构：每个子代理负责一个模块的修改
  • 代码审查：多个子代理分别审查不同方面（安全、性能、可读性）
  • 测试生成：并行为不同文件生成单元测试
  • 文档编写：子代理分别撰写 API 文档、README 和变更日志

示例 1：代码审查

$ ./target/debug/claw --model opus --permission-mode read-only > Review the authentication module for security vulnerabilities. Focus on: input validation, SQL injection, XSS, CSRF, and token handling.

示例 2：自动重构

$ ./target/debug/claw --model sonnet --permission-mode workspace-write > Refactor all API handler functions to use the new Result<T, AppError> return type. Update the error handling in each handler and add appropriate error codes. Run tests after each file change.

示例 3：批量分析

$ ./target/debug/claw --output-format json prompt \ "List all TODO comments in src/ with file, line, and context" \ | jq '.response' > todos.json

09 / Project Walkthrough

9. 实战案例：用 Claw Code 构建一个 REST API

以一个 Node.js + Express 的待办事项 API 项目为背景，演示如何在真实开发流程中使用 Claw Code 完成从项目初始化到部署验证的全过程。每个步骤都展示你发出的指令、代理的实际行为以及你需要关注的关键决策点。

步骤 1：项目初始化

先创建 CLAW.md 告诉代理项目的技术栈和规范，然后在 REPL 中发出脚手架指令。

# 1. 创建项目目录并编写 CLAW.md $ mkdir todo-api && cd todo-api && git init $ cat > CLAW.md << 'EOF' # CLAW.md ## Tech Stack - Node.js 20 + Express 4 + TypeScript - SQLite via better-sqlite3 (no ORM) - Jest for testing ## Conventions - Use ESM imports - All responses follow { data, error, meta } envelope - Commit messages: type(scope): description EOF # 2. 启动 Claw Code（workspace-write 模式保护系统文件） $ claw --model sonnet --permission-mode workspace-write # 3. 在 REPL 中发出指令 > Initialize a todo API project based on the tech stack in CLAW.md. Create package.json, tsconfig.json, src/index.ts with a health endpoint, and a basic Jest config. Run npm install after setup.
代理行为 Claw 会读取 CLAW.md 了解技术栈偏好，然后依次调用 WriteFile 创建配置文件、Bash 执行 npm install，并在完成后运行 npx tsc --noEmit 验证 TypeScript 配置是否正确。

步骤 2：实现 CRUD 端点

在同一个 REPL 会话中继续发出功能需求。上下文延续意味着代理已经理解了项目结构。

> Add CRUD endpoints for todos: - GET /api/todos (list, with ?status=done filter) - POST /api/todos (create, validate title is non-empty) - PATCH /api/todos/:id (update status) - DELETE /api/todos/:id Create a SQLite database module in src/db.ts and add migration logic that runs on startup. Write Jest tests for each endpoint.
代理行为 代理会创建 src/db.ts（数据库初始化 + 迁移）、src/routes/todos.ts（路由）、tests/todos.test.ts（测试），然后自动运行 npm test 验证。如果测试失败，代理会读取错误输出、诊断问题并自动修复——这就是 agentic loop 的价值。

步骤 3：安全审查

功能开发完成后，切换到 read-only 模式让代理审查代码，避免在审查过程中意外修改文件。

# 切换权限模式（在 REPL 内） > /permissions read-only # 发起安全审查 > Review all route handlers for security issues: 1. SQL injection in query parameters 2. Missing input validation 3. Error messages that leak internal details 4. Missing rate limiting considerations Output a numbered list of findings with severity and fix suggestions.
代理行为 代理会用 ReadFile 逐个读取路由文件，用 GrepSearch 搜索 SQL 拼接模式，然后输出结构化的审查报告。因为处于 read-only 模式，即使代理想修复问题也无法写入文件——你可以先审阅建议，再决定是否切回 workspace-write 让代理执行修复。

步骤 4：修复审查发现 & 提交

# 切回写入模式 > /permissions workspace-write # 让代理批量修复 > Fix all security findings from the review: - Use parameterized queries instead of string concatenation - Add input length validation (title max 200 chars) - Replace detailed error messages with generic ones in production Run tests after each fix to confirm nothing breaks. # 修复完成后，让代理生成 commit > Stage all changes and create a commit following our CLAW.md commit convention. Summarize what was fixed.
代理行为 代理会逐个修复问题，每次修改后运行测试确认不引入回归。最后执行 git add -A && git commit。整个流程中你只需要发出 3 条指令，代理完成了约 15 次工具调用。

步骤 5：CI 管道中的自动化检查

在 CI 环境中使用单次 prompt 模式进行无人值守的代码检查。

# 在 CI 脚本中（如 GitHub Actions） $ claw --model haiku --permission-mode read-only --output-format json \ prompt "Check if all API endpoints have corresponding test cases. \ Return JSON with { covered: [...], missing: [...] }" \ | jq '.response' > coverage-report.json # 检查是否有遗漏 $ jq '.missing | length' coverage-report.json 0
关键决策 CI 场景使用 haiku 模型（最快 + 最便宜）和 read-only 权限（防止 CI 环境中的意外写入）。JSON 输出让下游脚本可以解析结果并做出自动化决策。

10 / Best Practices

10. 最佳实践

从大量实际使用经验中总结出的高效使用方法。这些实践可以帮你减少 token 消耗、提高代理输出质量并避免常见陷阱。

01

明确指令优于模糊描述

❌「优化这个文件」→ 代理不知道优化什么。✅「将 src/db.ts 中的 SQL 字符串拼接改为参数化查询，保持现有的函数签名不变」→ 代理知道要改什么、怎么改、不改什么。指令越精确，代理的首次成功率越高，回合数和 token 消耗越低。

02

CLAW.md 是你最重要的投资

花 15 分钟写一份详细的 CLAW.md 可以节省数小时的反复纠正。包含：项目架构概览、命名规范、测试策略、错误处理约定和禁止事项。代理在每次会话开始时自动读取这份文件，相当于一次性培训了一个了解你项目的新同事。

03

要求代理先分析再动手

对于复杂任务，先让代理用 read-only 模式分析代码并输出计划，确认计划合理后再切换到 workspace-write 执行。这种「分析 → 确认 → 执行」的三步模式可以避免代理在错误方向上浪费大量 token。

04

按任务粒度拆分会话

一个 REPL 会话最好围绕一个聚焦目标。完成一个功能后用 /clear 开始新会话，而不是在同一个会话里塞下整个 sprint 的工作。上下文越短，代理的注意力越集中，输出质量越高。当会话变得臃肿时，用 /compact 压缩而不是继续累加。

05

为不同任务选择合适的模型

opus 用于需要深度推理的任务（架构设计、复杂重构、安全审计）；sonnet 用于日常开发（功能实现、Bug 修复、代码生成）；haiku 用于批量操作（格式检查、TODO 收集、CI 检查）。不要为简单任务使用 opus——费用是 sonnet 的数倍但对简单任务没有质量提升。

06

让代理自验证

在每条指令中明确要求「修改后运行测试」或「编译验证」。代理天然支持 agentic loop——如果测试失败，它会自动读取错误、诊断并重试。你要做的是在指令中设定验证标准，而不是手动检查每一步。如果代理连续失败 3 次以上，这通常意味着需求不够清晰或超出了模型能力，此时应该介入调整指令。

07

权限模式是你的安全网

默认从 workspace-write 开始，而不是 danger-full-access。只有在你完全信任代理行为（如个人项目）或在隔离环境中（如容器化 CI）时才升级到 full-access。代码审查一律使用 read-only。养成在 REPL 中随时用 /permissions 检查当前模式的习惯。

08

配置分层管理

团队项目中，在 .claw.json 设置保守默认值（sonnet + workspace-write），提交到 git。让每个人在 settings.local.json 中覆写个人偏好（如 opus + extended-thinking）。这样既保证了团队基线安全，又不限制个人的工作方式。settings.local.json 必须加入 .gitignore。

09

善用 /compact 而非 /clear

当上下文变长但你还需要保留核心上下文时，用 /compact 而不是 /clear。/compact 会压缩对话但保留关键信息（如当前正在修改的文件列表、已做的决策）。只有在切换到完全不相关的任务时才用 /clear。用 /status 监控 token 用量，在接近限制前主动压缩。

10

按文件边界划分会话

多会话并行时，最有效的冲突预防策略是让每个会话只修改不重叠的文件集合。例如会话 A 负责 src/auth/，会话 B 负责 src/api/。如果两个会话必须触及同一文件，应让其中一个先完成并提交，另一个再开始。在 CLAUDE.md 中写明「本会话只修改以下文件：…」可以显式约束代理的操作范围。

11

Git Worktree 隔离并行任务

git worktree 是多会话协作的最佳拍档。为每个并行任务创建独立 worktree 和分支（git worktree add ../feature-x -b feature-x），每个会话在各自 worktree 的独立目录中运行 Claw Code。这样即使两个代理同时修改相同路径的文件也不会冲突，因为它们操作的是物理上不同的目录。最后通过 PR 合并，利用 code review 解决语义冲突。

12

频繁提交，小步合并

每完成一个逻辑完整的修改就让代理提交（在指令中加「完成后 git commit」）。小而频繁的提交使合并窗口更短、冲突范围更小、回滚更精确。避免让一个会话积累数小时的未提交修改——积累越久，与其他会话产生文本冲突和语义冲突的概率越高。合并前用 git diff main...feature-x 预览差异，确认影响范围。

13

用 CLAUDE.md 建立会话契约

在多人或多会话场景中，CLAUDE.md 不仅是配置文件，更是协作契约。写入共享的编码规范（缩进、命名、commit message 格式）、架构约束（不引入新依赖、不修改公共接口）和测试要求（修改后必须通过 lint 和测试）。这保证不同会话产出的代码风格一致，减少合并后的整理成本。每个 worktree 可以有自己的 CLAUDE.md 来进一步限定该分支的工作范围。

11 / Troubleshooting

11. 常见问题

收集 Claw Code 使用过程中的常见问题及其解决方案。

症状：启动后提示认证错误或 401 响应。

解决：确认 ANTHROPIC_API_KEY 环境变量已正确设置，或尝试 claw login 进行 OAuth 认证。检查 Key 是否已过期或被吊销。如果使用代理，确认 ANTHROPIC_BASE_URL 指向正确的端点。

症状：cargo build --workspace 失败。

解决：确认 Rust 版本足够新（建议 stable 最新版）。尝试 rustup update 更新工具链。清理缓存后重试：cargo clean && cargo build --workspace。

症状：代理尝试写入文件或执行命令时被阻止。

解决：检查当前权限模式（/permissions）。如果在 read-only 模式下需要写入，使用 /permissions 或重启并指定 --permission-mode workspace-write。路径穿越检测触发时，确认操作目标在工作区目录内。

症状：代理响应变得不相关或提示 token 超限。

解决：使用 /compact 压缩会话上下文，保留关键信息并释放空间。如果仍然不够，使用 /clear 清空历史重新开始。对于大型任务，建议拆分为多个独立的子会话。

症状：MCP 工具调用返回错误或超时。

解决：Claw Code 的 MCP 设计支持部分成功——其他 MCP 服务器和内置工具不受影响。检查目标服务器是否在运行、网络是否可达，以及认证凭证是否正确。
