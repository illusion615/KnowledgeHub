Claw Code 深度解析：自主编码代理的开源 Harness 实现

Claw Code 是由 UltraWorkers 社区维护的开源编码代理 harness，其核心理念是用自主代理来构建自主代理——代码本身由 AI 代理协同编写、测试和部署。项目已从 TypeScript 移植到 Python，当前主力实现为 Rust，包含 9 个 crate、约 48,600 行 Rust 代码、40 个工具定义和完整的 mock 对齐测试体系。

Claw Code 不是传统的开发工具，而是一个由三层自治系统驱动的编码 harness：工作流层编排任务，事件层路由通知，协作层解决分歧。人类角色被简化为一句话指令输入。

01 / Project Overview

1. 项目总览

Claw Code 是 ultraworkers/claw-code 仓库的开源编码代理 harness。它最初以 TypeScript 编写，随后被移植为 Python，当前主力实现是 Rust。项目的独特之处在于：代码本身是由自主编码代理协同构建的。

01

背景

由 Bellman / Yeachan Heo 和 Yeongyu 发起，项目通过 clawhip、oh-my-openagent、oh-my-claudecode、oh-my-codex 等自主编码工作流推进。仓库在发布 2 小时内即突破 50K star。

02

移植历程

TypeScript → Python (src/) → Rust (rust/)。当前 Rust 占比 95.6%，Python 4.4%。Python 层保留了 port manifest 和 parity audit 查询引擎，用于跟踪移植进度。

03

仓库统计

292 commits on main，9 个 Rust crate，48,599 行 Rust 代码，2,568 行测试代码，3 位作者，开发周期 2026-03-31 → 2026-04-03。

04

动机

作者研究了原始代码的 harness 和工具连接方式后，出于法律与伦理考量决定不保留原始快照，转而进行独立的移植重写工作。这不是简单的抄袭，而是基于理解的重新实现。

02 / Philosophy

2. 核心哲学：人类定方向，Claw 做执行

Claw Code 的核心观点是：当编码智能足够便宜且广泛可用时，稀缺资源不再是打字速度，而是架构清晰度、任务分解能力、判断力和产品品味。人类的工作不是超越机器的编码速度，而是决定什么值得被构建。

传统模式

人类驱动的开发

开发者在终端中逐行编码、手动运行测试、通过 PR review 协作。AI 作为辅助工具提供补全建议。瓶颈是人的打字速度和上下文切换能力。知识在会话间丢失，重复劳动不可避免。

Claw 模式

自主代理驱动的开发

人类通过 Discord 发送一句话指令，AI 代理并行分解任务、编写代码、运行测试、解决冲突并推送结果。人可以走开、睡觉或做其他事情。瓶颈转移到了方向判断和架构设计。

在 Claw Code 的工作流中，关键的人类接口不是 tmux、Vim 或 SSH，而是 Discord 频道。一个人可以在手机上输入一句指令，然后走开。Claw 读取指令，分解为任务，分配角色，编写代码，运行测试，在失败时争论，恢复，并在工作通过时推送。

当代理系统能在数小时内重建一个代码库时，稀缺资源变成了：架构清晰度、任务分解能力、判断力、品味、关于什么值得构建的信念，以及知道哪些部分可以并行化、哪些必须保持约束。快速的代理团队不会消除思考的需要，反而让清晰思考变得更有价值。

不要只盯着生成的文件——它们只是错误的层级。Python 重写是副产品，Rust 重写也是副产品。真正值得研究的是产生它们的系统：一个基于 clawhip 的协调循环，人类提供方向，自主 claw 执行工作。仓库是产出物，协调系统背后的哲学才是教训。

03 / System Architecture

3. 三层协作系统架构

Claw Code 的自主开发能力依赖三个相互独立但协同工作的系统层。每一层解决自主编码中的不同维度问题：工作流编排、事件路由和多代理协作。

OmX (oh-my-codex)

工作流层。将短指令转化为结构化执行：规划关键词、执行模式、持久验证循环、并行多代理工作流。把一句话变成可重复的工作协议。

任务编排

clawhip

事件和通知路由器。监控 git commit、tmux session、GitHub issues/PR、代理生命周期事件和频道投递。将监控和通知推出编码代理的上下文窗口，让代理专注实现。

事件路由

OmO (oh-my-openagent)

多代理协调层。处理规划、任务交接、分歧解决和跨代理验证循环。当 Architect、Executor 和 Reviewer 意见不一致时，OmO 提供收敛结构而非发散。

多代理协调
设计关键 clawhip 最重要的设计决策是将通知路由从代理上下文窗口中剥离。编码代理不应该浪费 token 在格式化状态报告和路由通知上——这些是事件层的责任。

04 / Rust Workspace

4. Rust Crate 工作区详解

当前 Rust 实现由 9 个 crate 组成，按职责清晰划分为 API 通信、运行时逻辑、CLI 交互、工具实现等层级。以下是各 crate 的职责和关键实现。

api

HTTP 客户端、SSE 流解析、请求/响应类型、认证（API Key + OAuth Bearer）

runtime

ConversationRuntime 代理循环、ConfigLoader 层级配置、Session 持久化、权限策略、MCP 客户端、系统 prompt 组装、用量追踪

tools

40 个工具规格定义与执行实现：Bash、ReadFile、WriteFile、EditFile、GlobSearch、GrepSearch、WebSearch、WebFetch、Agent、TodoWrite 等

rusty-claude-cli

主 CLI 二进制（claw 命令）：REPL 交互、单次 prompt、流式输出显示、工具调用渲染、参数解析

commands

Slash 命令定义和帮助文本生成：/status、/compact、/clear、/model、/permissions、/config 等

plugins

插件元数据、注册表和 Hook 集成面：install/enable/disable/uninstall 生命周期管理

mock-anthropic-service

确定性 /v1/messages mock 服务，用于 CLI 对齐测试和本地 harness 运行

compat-harness

从上游 TypeScript 源码提取工具/prompt 清单，用于对齐验证

telemetry

Session 追踪事件和遥测载荷类型定义

Anthropic API + streaming ✅ OAuth login/logout ✅ Interactive REPL ✅ Tool system (bash/read/write/edit/grep/glob) ✅ Web tools (search/fetch) ✅ Sub-agent orchestration ✅ Todo tracking ✅ Notebook editing ✅ CLAUDE.md / project memory ✅ Config file hierarchy ✅ Permission system ✅ MCP server lifecycle ✅ Session persistence + resume ✅ Extended thinking ✅ Cost tracking ✅ Git integration ✅ Model aliases (opus/sonnet/haiku) ✅ Slash commands ✅ Hooks (PreToolUse/PostToolUse) 🔧 Plugin system 📋 Skills registry 📋

CLI 支持三级权限模式，通过 PermissionEnforcer 在工具执行前进行守门检查：

  • read-only — 只允许读取操作，bash 中的写入命令被拒绝
  • workspace-write — 允许在工作区边界内写入文件，跨越工作区目录的操作被阻止
  • danger-full-access — 完全访问权限，默认模式

运行时配置按以下顺序加载，后项覆盖前项：

  1. ~/.claw.json
  2. ~/.config/claw/settings.json
  3. <repo>/.claw.json
  4. <repo>/.claw/settings.json
  5. <repo>/.claw/settings.local.json

06 / Toolchain Ecosystem

6. 工具链生态：UltraWorkers 套件

Claw Code 不是一个孤立的项目，它是 UltraWorkers 生态系统的核心产出。理解完整生态有助于理解为什么 Claw Code 的代码结构看起来不像人类手写。

01

clawhip

事件和通知路由器。监控 git、tmux、GitHub、代理生命周期事件。让编码代理不必在上下文窗口中处理状态汇报，专注于代码实现。

02

oh-my-codex (OmX)

工作流编排层。提供 $team 模式（协调并行审查）和 $ralph 模式（持久执行 + 验证纪律）。将短指令转化为结构化并行工作流。

03

oh-my-openagent (OmO)

多代理协调框架。当 Architect、Executor、Reviewer 三个角色意见不一致时，提供收敛结构避免循环发散。

04

oh-my-claudecode

Claude Code 的增强层。提供代理级封装，使 Claude Code 可以被 clawhip 和 OmX 调度管理，参与自主编码循环。

07 / Evolution Roadmap

7. 演进路线：从 CLI 到 Claw-Native 运行时

Claw Code 的目标是从「一个人类也能用的 CLI」演进为「Claw-native 执行运行时 + 事件原生编排底座 + 插件/Hook 优先的自主编码 harness」。路线分为五个阶段。

1

可靠 Worker 启动

就绪握手生命周期、信任提示自动解析、结构化会话控制 API

2

事件原生集成

标准车道事件 Schema、失败分类法、可操作摘要压缩

3

分支/测试感知

陈旧分支检测、绿色度契约、常见失败自动恢复

4

Claw 优先执行

类型化任务包格式、自主编码策略引擎、机器可读车道看板

5

插件/MCP 成熟

一等插件/MCP 生命周期契约、端到端 MCP 对齐、降级模式报告

一个 clawable 的 harness 需要满足七个条件：

  • 启动确定性 — 每次启动行为一致
  • 状态和失败模式机器可读
  • 可在无人监控终端时恢复
  • 分支/测试/worktree 感知
  • 插件/MCP 生命周期感知
  • 事件优先而非日志优先
  • 可自主执行下一步
  1. 状态机优先 — 每个 worker 有显式生命周期状态
  2. 事件优于刮取 — 频道输出从类型化事件派生
  3. 恢复先于升级 — 已知失败模式先自动修复一次
  4. 分支新鲜度先于归责 — 检测陈旧分支后再判断是否为新回归
  5. 部分成功是一等公民 — MCP 启动允许部分服务器成功
  6. 终端是传输层不是真相源 — 编排状态必须在 tmux/TUI 之上
  7. 策略可执行 — 合并/重试/rebase/清理规则由机器执行

08 / Key Takeaways

8. 关键启示与思考

Claw Code 不仅是一个编码工具——它是自主软件开发的一个公开实验。从这个仓库中可以提取出对企业级代理架构设计有价值的模式和启示。

01

关注点分离模式

通知路由从代理上下文窗口分离（clawhip），工作流编排从代码执行分离（OmX），冲突解决从任务分配分离（OmO）。每一层只做一件事。

02

对齐即测试

通过 mock parity harness 实现确定性端到端验证——不依赖真实 API，而是用脚本化场景确保行为一致性。这比传统的集成测试更适合代理系统。

03

Lane 并行模式

9 条独立 lane 并行开发，每条 lane 有明确的 commit hash、合并状态和影响范围。这种组织方式允许多个代理同时推进不同功能模块，互不干扰。

04

代理即一等开发者

Claw Code 的 Roadmap 中最深刻的转变是：从「为人类设计 CLI，代理适配使用」到「为代理设计执行运行时，人类通过 chat 接口提供方向」。终端是传输层而非真相源。

05

自举悖论

用代理来构建代理工具本身是一个自举实验。Claw Code 在 3 天内产出 48,600 行 Rust 代码和 292 个 commit，验证了自主编码工作流在高速迭代场景中的可行性，同时也暴露了 CI 稳定性和文档一致性的挑战。

06

开源伦理实践

项目明确声明不持有原始 Claude Code 源材料的所有权，也不隶属于 Anthropic。在研究原始 harness 后选择独立重写而非保留原始快照，体现了对逆向工程和 copyleft 问题的慎重态度。
