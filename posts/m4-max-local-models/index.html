Apple Silicon Local AI Stack

MacBook Pro M4 Max 64GB 本地模型部署选型：LLM、图像、视频、语音与音乐生成全景

对 M4 Max 64GB 这类 Apple Silicon 笔记本而言，本地部署的关键瓶颈不是“能不能跑模型”，而是“哪一类模型有成熟 Metal/MPS 路径、能在 64GB 统一内存内稳定迭代、并且延迟仍然适合日常工作”。本文把选型拆成六类任务：大语言模型、Function Calling、文生图、视频生成、语音生成和音乐生成，并按 内存占用、运行时成熟度、Apple 兼容性、交互体验 四个维度比较，给出适合 M4 Max 的推荐组合。

说明：文中“性能”优先指 资源级性能，即模型在 64GB 统一内存里的可放置性、是否需要频繁 swap、以及是否适合交互式使用。凡是出现具体时间或显存数字，都明确标注其来源条件；未在本文环境中实测的指标，不伪造本机速度。

核心结论

M4 Max 64GB 可以胜任高质量本地 LLM、图像与轻量音频生成，但视频仍然属于“可试验，不宜作为主力生产”。

01 / Baseline & Constraints

1. 先看机器边界：64GB 统一内存决定了什么能常用，什么只能偶尔跑

Apple Silicon 的优势是统一内存和 Metal 生态，弱点是大量开源生成模型默认以 CUDA 为第一目标。对本次选型，先定义三个判断条件：能否稳定装下、有没有成熟 Apple 运行栈、生成时长是否适合日常交互。其中第一条最重要，因为一旦进入 swap，Mac 的体感会从“可用”直接滑落到“不可持续”。

Criterion 01

内存占用先于峰值质量

64GB 并不意味着可以无成本常驻 70B。LLM 除权重外还要叠加 KV cache、上下文和采样缓冲；扩散模型则要叠加文本编码器、VAE、中间 latent 和 attention 工作区。

Criterion 02

优先选择 Apple 原生或 Metal 一等公民栈

LLM 首选 MLX-LM，其次是 GGUF + Metal 的 llama.cpp；多模态首选 Hugging Face Diffusers 的 MPS 路线。CUDA 专属优化越重，迁移到 Mac 的折损越大。

Criterion 03

把“能跑”与“适合工作”分开

能完成一次生成不等于适合常用。本文把结果分为三档：交互级、批处理级、实验级。交互级适合频繁往返调参；实验级只建议跑样片或验证可行性。

Weight Math

8B 级

4-bit 权重的理论下限约为 8B × 0.5 byte ≈ 4GB。考虑量化元数据、张量布局和运行时开销，日常工作集更接近 5GB 到 7GB。

Weight Math

14B 级

理论权重约 7GB，实际常见工作集更接近 9GB 到 12GB。这个区间是 M4 Max 64GB 最适合的高质量常驻档。

Weight Math

27B - 32B 级

4-bit 理论权重约 13.5GB 到 16GB，实际工作集容易进入 16GB 到 26GB。可以跑，但上下文一长、并发一多，交互感会明显变差。

Weight Math

70B 级

4-bit 理论权重约 35GB，算上缓存后非常容易挤压到系统可用内存边界。对于笔记本形态，不建议作为本地主力。

结论：在 M4 Max 64GB 上，模型选型不该围绕“最大的参数量”，而应围绕“14B 级是否已经满足任务质量”。对多数中文助手、工具调用、知识整理和原型生成场景，答案是肯定的。

02 / LLM & Tool Use

2. LLM 与 Function Calling：Qwen3 和 GLM-4 重新定义了本地选型格局

Apple 自家的 MLX-LM 已经覆盖 Qwen3 全系列[1]。llama.cpp 的 Metal 后端同样成熟[2]。在工具调用层，Ollama 已提供单/并行/agent loop 文档[3]。Qwen3 官方明确推荐 Ollama、LM Studio、MLX 和 llama.cpp 作为本地方式[4]；GLM-4-32B-0414 在 llama.cpp 中已有合并的模型实现[17]；Llama 4 Scout/Maverick 虽然发布了 MoE 架构，但 109B+ 总参对 64GB 笔记本仍然偏重[18]。

Qwen3 在 2025 年 4 月发布，相比 Qwen2.5 有三个关键升级：(1) thinking/non-thinking 混合模式，可按任务复杂度动态切换推理深度；(2) 预训练数据从 18T 扩至 36T tokens，覆盖 119 种语言；(3) 同级别模型质量大幅提升 — Qwen3-4B 匹敌 Qwen2.5-72B，Qwen3-30B-A3B（MoE，3B 激活）超越 QwQ-32B。官方明确推荐 Ollama、LM Studio、MLX 和 llama.cpp 作为本地运行方式[4]。

Gemma 4 是 Google 基于 Gemini 3 技术构建的最新开源模型，26B/31B 两档面向工作站和编码助手。原生支持 function calling、多模态（视觉 + 音频）和 140 语言[5]。在 AIME 2026 上达到 88.3%，τ2-bench agentic tool use 达到 85.5%，已上架 Ollama 和 LM Studio。26B 的内存占用与 Gemma 3 27B 接近但能力有代际提升。

若目标是开发速度，优先 Ollama，因为它已内置单工具、并行工具与 agent loop 样例[3]。若目标是协议控制和 JSON 约束，优先 llama.cpp 的 OpenAI 兼容 server 与 grammar。若目标是 Apple 原生吞吐，则用 MLX-LM 跑模型，把工具协议留给上层服务适配。

03 / Multimodal Matrix

3. 多模态模型对比：图像和音频已经实用，视频仍然明显偏向实验

Hugging Face Diffusers 已经把 Apple Silicon 的 MPS 作为正式优化路径，官方建议在内存紧张时打开 attention slicing，并提醒 batch 生成在 MPS 上可能不稳定[6]。这给图像和部分音频模型提供了可落地路径；但视频模型的主流优化仍围绕 CUDA、TorchAO、FSDP 或多卡展开，Mac 兼容层远不如图像成熟[8][9]。

04 / Model Engines

4. 模型运行引擎对比：Ollama、oMLX、LM Studio、MLX-LM、llama.cpp、Jan 和 GPT4All

"模型"和"跑模型的引擎"是两回事。同一个 Qwen3-14B，用 MLX-LM 跑和用 oMLX 跑，速度、功能和开发体验可以差一个等级。这一节把七个最常见的本地引擎 / 平台放在一起比较，维度是：推理性能、Apple Silicon 兼容性、功能丰富度（API、工具调用、RAG、MCP）和社区活跃度。

引擎 类型 推理性能 Apple 兼容性 功能丰富度 社区活跃度 M4 Max 建议定位
Ollama CLI 服务 + API 高 高 高 高 默认首选。底层基于 llama.cpp，开箱即用的 OpenAI 兼容 API、工具调用（单/并行/agent loop）、流式输出[3]。社区最大、教程最多。
LM Studio GUI + CLI + API 高 高 高 高 可视化首选。同时支持 GGUF (llama.cpp) 和 MLX 双后端，MCP 集成、JS/Python SDK、文档对话 RAG、headless 部署 (llmster)、speculative decoding[14]。
MLX-LM Python 库 / CLI 高 最高 中 中 Apple 原生吞吐最优。Apple 官方团队维护，量化、长上下文缓存、wired memory 调优原生支持[1]。功能聚焦于推理与模型转换，没有内置 GUI 或 API server。
oMLX macOS App + CLI + API 高 最高 很高 高 MLX 原生推理服务化首选。基于 mlx-lm 构建，加入 continuous batching、SSD 分级 KV cache、多模型并存、菜单栏管理与 OpenAI/Anthropic 兼容 API[19]。支持 tool calling、VLM、embedding 和 MCP。是"把 MLX-LM 变成生产级推理服务器"的最佳方案。
llama.cpp C++ 库 / CLI / Server 高 高 高 很高 底层基石。Metal 为一等公民后端，OpenAI 兼容 server、grammar 约束、speculative decoding、多模态[2]。是 Ollama 和 LM Studio 的推理后端之一。
Jan GUI 客户端 + API 高 高 高 中高 终端用户首选。开源（AGPL），500 万+下载，内置 web search、MCP、远程 provider 接入、多语言 UI[15]。同时支持 GGUF 和 MLX（社区测试 MLX q4 达 90 tok/s vs llama.cpp 49 tok/s）。
GPT4All GUI 客户端 中 中 中 中 全离线入门。LocalDocs RAG 可直接对本地文件问答[16]。但推理栈非 MLX 原生，没有 tool calling API，不适合做开发级集成。

如果你主要需要 API 接入和自动化，Ollama 更适合 — 它天生是一个 daemon 服务，开机后台运行，任何脚本或应用直接调用 REST 接口[3]。如果你更想要可视化聊天、参数调节和模型管理界面，LM Studio 是更好的选择。LM Studio 0.4 以后还支持 headless 模式 (llmster)，可以纯粹当 server 用[14]。两者都基于 llama.cpp 后端，切换成本极低。

MLX-LM 是底层库：Apple 官方团队维护的 Python 推理和量化框架[1]。它提供模型加载、生成、量化和提示缓存等核心能力，但不包含 API server、多模型管理或 GUI。

oMLX 是基于 mlx-lm 构建的服务化层：它在 mlx-lm 的推理核心之上加入了 continuous batching、block-based 分级 KV cache（RAM 热层 + SSD 冷层）、多模型并存 + LRU 自动管理、macOS 菜单栏原生 App、OpenAI/Anthropic 兼容 API、VLM/embedding/reranker 支持、tool calling 和 MCP 集成[19]。

选择逻辑：如果你只需要在 Python 脚本里调用 MLX 跑一次推理或做模型量化，直接用 MLX-LM。如果你需要把 MLX 模型作为常驻服务暴露 API 给多个客户端（如 Claude Code、Copilot、自定义 agent），让多个模型共存并自动管理内存，那 oMLX 就是 MLX-LM 的"生产级壳"。二者不冲突 — oMLX 的推理路径就是 MLX-LM。

MLX-LM 是 Apple 官方的 ML 框架上层库，推理路径直接走 Metal shader graph，不经过 GGUF / GGML 转换[1]。在 Apple Silicon 上，MLX 通常比 llama.cpp 的 GGUF 路线在吞吐上更快（社区测试 MLX q4 达到 90 tok/s vs llama.cpp 49 tok/s），但模型格式生态不如 GGUF 丰富。适合"确定了用什么模型，追求跑得最快"的场景。

Jan 的定位是"个人 AI 客户端"，内置 web search、MCP 和多 provider 接入（OpenAI、Claude、Gemini、HuggingFace 等）[15]。如果你想要一个"类 ChatGPT 但完全本地"的体验，不想折腾命令行和 API，Jan 是最好的起点。但如果你要做 agent 或自动化管线，Ollama 和 llama.cpp 的工程控制力更强。

GPT4All 的最大卖点是全离线和 LocalDocs RAG[16]。但它不提供 tool calling API，推理后端不支持 MLX 原生加速，模型更新速度和社区活跃度也落后于 Ollama 和 LM Studio。在 M4 Max 上，它更适合当"不想碰命令行的备用聊天窗口"，不适合当开发栈。

Dev

开发者：oMLX 或 Ollama

如果你主要跑 MLX 格式模型，oMLX 是最强选择 — continuous batching、SSD KV cache 和 Claude Code 优化都是原生 MLX 路线独有的。如果你需要跑 GGUF 格式或追求最广的社区教程覆盖，Ollama 仍是默认首选。

Power User

高阶用户：LM Studio

同时支持 GGUF 和 MLX 双引擎、可视化调参、MCP 集成、headless 部署。是功能覆盖面最广的单一平台。

Casual

轻度用户：Jan

界面最友好、开箱即用、内置搜索和多 provider。不需要折腾命令行就能用本地模型替代 ChatGPT 和 Perplexity。

05 / Runtime Stacks

5. 推荐运行栈：不要试图用一套框架统一所有任务

本地模型部署在 Mac 上最常见的误区，是试图用一种框架覆盖 LLM、图像、视频和音频。实际上更合理的方式是按任务拆栈：LLM 用 MLX 或 GGUF；图像和音频用 Diffusers/MPS；视频只在有明确需求时单独搭建实验环境。

Stack A

MLX-LM

  • 最适合 Apple Silicon 上的 LLM 推理与量化。
  • 官方支持模型转换、量化、长上下文缓存和大模型 wired memory 调优[1]。
  • 适合作为“核心推理引擎”，但工具协议最好交给上层服务处理。
Stack B

llama.cpp / GGUF

  • Metal 是一等公民后端，OpenAI 兼容 server 与 grammar 支持完整[2]。
  • 适合追求模型兼容性、轻量 API 服务、JSON 约束和多模型切换。
  • 如果要跑很多社区量化模型，它往往比 MLX 更省心。
Stack C

Ollama

  • 最适合快速上手 agent loop、并行工具调用和 OpenAI 兼容接入[3]。
  • 适合把本地模型包装成统一 API，不适合承担所有底层优化。
  • 建议把它放在应用层，而不是当作唯一推理栈。
Stack D

Diffusers + PyTorch MPS

  • 图像、音频和部分视频任务的主通道。
  • 官方明确支持 Apple Silicon，但提醒 attention slicing、batch 不稳定和大张量限制[6]。
  • 适合做多模态实验环境，不建议和主 LLM 环境混装在同一个 Python 解释器里。
1

日常助手栈

Qwen3-14B 4-bit + MLX-LM。若应用要用 OpenAI 风格函数调用，在外面包一层 Ollama 或 llama.cpp server。

2

图像创作栈

SDXL 走 ComfyUI 或 Diffusers/MPS 打基础；FLUX.1-schnell 作为高质量补充。两者分环境安装，避免 PyTorch、xformers、社区插件互相污染。

3

音频栈

Kokoro 做常规 TTS，MusicGen 做短音乐草稿，Stable Audio Open 做环境声和音效。F5-TTS 只在确实需要更强表现力时单独启用。

4

视频栈

单独隔离为实验环境。优先尝试 Wan2.1 1.3B 或 CogVideoX-2B，不把它们并入主工作流，也不把视频生成的成功率当作整机 AI 能力的判断依据。

06 / Recommendations

6. 最终建议：先把 Qwen3-14B、SDXL 和 Kokoro 跑顺，再决定是否投入 FLUX 与视频

如果目标是“在 M4 Max 64GB 上建立一套长期可维护的本地 AI 环境”，最优策略不是一次装满所有热门模型，而是分三层推进：先完成主力任务，再扩展高质量图像，最后才评估视频。下面给出三种面向不同目标的组合建议。

Default Build

默认工作组合

  • LLM：Qwen3-14B 4-bit
  • Function Calling：Ollama 或 llama.cpp server
  • 文生图：SDXL
  • 语音：Kokoro
  • 音乐：MusicGen-medium

这是综合稳定性、中文能力、资源占用和部署难度后的最佳平衡点。

Quality Build

质量优先组合

  • LLM：Qwen3-32B 或 GLM-4-32B 离线任务 + Qwen3-14B 常驻
  • 文生图：FLUX.1-schnell
  • 语音：F5-TTS
  • 音乐：Stable Audio Open 1.0

适合愿意接受更长生成时间、并愿意为环境隔离投入更多工程成本的用户。

Research Build

研究实验组合

  • 视频：Wan2.1 T2V-1.3B 或 CogVideoX-2B
  • 图像：SDXL 作为对照基线
  • LLM：Gemma 4 26B 或 Qwen3-14B 负责 prompt 扩展

适合验证视频与多模态链路，不适合作为日常生产环境。

A

如果你主要做本地知识助手与 Agent

直接选 Qwen3-14B。它在中文、JSON、结构化输出和工具调用上的综合平衡，加上 thinking/non-thinking 混合模式，是 M4 Max 64GB 上最值得长期使用的主力模型。若 function calling 是核心需求，额外测试 GLM-4-32B-0414。

B

如果你主要做图像创作

先上 SDXL，确认 MPS、VAE、LoRA 和工作流节点都跑顺，再追加 FLUX.1-schnell。不要从最重的模型开始，否则你会把时间消耗在环境对齐而不是产出上。

C

如果你想做视频

结论很直接：M4 Max 64GB 可以研究视频生成，但不应该被当作视频生成主机。若视频是高频生产任务，迁移到有大显存 NVIDIA 的桌面机或云端会更合理。
